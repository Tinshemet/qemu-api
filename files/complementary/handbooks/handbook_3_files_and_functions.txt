================================================================================
  GORGON  —  FILES AND FUNCTIONS REFERENCE  (handbook part 3)
================================================================================

Every source file is documented with its purpose, its public functions, what
each function does, its inputs and outputs. Config keys are in handbook part 4.

Three-tier layout: orchestrator (AI + HTTP gating) -> executor (owns QEMU/KVM,
can run as its own HTTP service) -> client (thin remote UI). shared/ holds code
used by more than one tier. admin/ is a standalone curses dashboard that talks
to the orchestrator over HTTP like any other client.

────────────────────────────────────────────────────────────────────────────────
orchestrator/ai/cli.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The interactive AI chat REPL entry point. Loads session state, prints the
  banner, runs the agentic tool loop (up to a configurable iteration cap per
  user turn), and handles the small set of instant local slash-commands
  (list/system/profiles/clear-session/drift/auto-clear/loop-limit). This is
  the module __main__ resolves to when invoked with no CLI arguments; with
  arguments it hands off to direct_cli.cli_direct(). There is no more
  "ollama_wrapper.py" shim — this file (plus direct_cli.py) IS the entry
  point package, imported directly as orchestrator.ai.cli.

MODULE-LEVEL STATE
  _CFG            — config from orchestrator/ai/config.json
  _EXIT_CMDS      — set of strings that exit the chat loop
  _SHORTCUTS      — dict of instant-response commands (list, profiles, etc.)
  _LOOP_MAX       — max consecutive AI tool-call iterations per user turn
                    (get_loop_max() — respects a persisted override)
  _ACTION_WORDS / _STATE_QUERY_WORDS — words that mark a user turn as
                    "wants a tool" (both mutating actions and read/state
                    queries), so factual questions get grounded in a tool
                    call instead of answered from the model's memory
  _OS_KEYWORDS    — OS-type keywords used by the os_type resolver
  _CONFIRM_YN / _CONFIRM_NAME — per-tool safety-confirmation shape
  _RENDERS_OUTPUT — tools whose result is already rendered to the user (the
                    AI is told not to repeat it in text)

FUNCTION: chat_loop(verbose=False)
  IN:  verbose bool
  OUT: does not return (runs until user exits)
  1. Prints the startup banner (OVMF status, Ollama URL/model, executor mode).
  2. Auto-clears the session if configured; loads it otherwise.
  3. Runs detect_drift() and reports/auto-clears on a critical drift signal.
  4. If executor_client.API_URL != "local": starts a background liveness
     thread that pings {API_URL}/health every 30s and warns on failure.
  5. REPL loop: reads a line, checks _handle_command() shortcuts, then drives
     the agentic tool loop — calls _call_ollama(), and for each tool_call in
     the response routes it through chat_turn._process_tool_call() (context
     assistant -> os_type resolution -> preflight -> safety confirm -> manual
     config prompt -> execute -> clarify drain). Handles GateOutcome.EXIT /
     REPLAN / CANCELLED / SKIP_TOOL to decide whether to keep looping, break,
     or return from chat_loop entirely.
  6. Tracks runtime drift (consecutive action turns with no tool executed)
     and warns/escalates via console messages.
  7. Persists the session via save_session() after each turn.

FUNCTION: _handle_command(ui, messages, runtime_drift_count, verbose) -> bool
  IN:  ui str (lowercased user input), messages list[dict], runtime_drift_count
       int, verbose bool
  OUT: bool — True if the input was a recognised shortcut and was handled
  Covers: list [label] / system / profiles / clear-session / drift /
  auto-clear on|off / loop-limit [n]. Returns False (falls through to the AI)
  for anything else.

FUNCTION: _show_drift_report(messages, runtime_drift_count)
  IN:  messages list[dict], runtime_drift_count int
  OUT: nothing (console output)
  Renders a Rich panel: session message counts, orphaned-turn ratio, max
  consecutive orphans, and detect_drift()'s health verdict.

KEY HELPERS
  __main__ block  — parses -v/--verbose, -cu (custom mode), -cs (clear
                    session first) from sys.argv, then calls direct_cli.
                    cli_direct(argv) if there are remaining args, else
                    chat_loop(verbose).

IMPORTANT NOTES
  - Re-exports process_message (from .http_chat) and TurnState/GateOutcome/
    _process_tool_call/_build_vm_spec_rows (from .chat_turn) so other modules
    (api_server.py, tests) can import them from cli without reaching into the
    split-out files directly.
  - The old monolithic server/ai/cli.py has been split four ways: this file
    (REPL + shortcuts), direct_cli.py (non-interactive sub-commands),
    chat_turn.py (the five interactive gates), chat_types.py (gate-independent
    base types), and http_chat.py (the HTTP single-turn processor). Imports
    are one-directional: cli -> chat_turn -> chat_types, and cli -> http_chat
    -> chat_turn's _is_critical only; http_chat never imports cli.

────────────────────────────────────────────────────────────────────────────────
orchestrator/ai/direct_cli.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Dispatches the non-interactive `gorgon <cmd>` sub-commands (list, launch,
  stop, snapshot, network, serve, fetch, setup-done, guest-setup, …) straight
  to the manager or executor_client and renders the output with Rich. No
  Ollama call is made for any of these. Imported by cli.py's __main__ only
  when argv is non-empty; this module never imports from cli.py itself (the
  edge is one-directional, so there's no import cycle).

FUNCTION: cli_direct(args, verbose=False)
  IN:  args List[str] (argv with flags already stripped), verbose bool
  OUT: nothing (renders directly to console)
  Big dispatch on args[0]: list / status / monitor / launch / stop / config /
  resize / clone / snapshot / network / limit / delete / cmd / profiles /
  templates / check-profile / system / isos / show-cmd / setup-done /
  guest-setup / fetch / serve / clear-session / -tf. Falls back to a rendered
  help panel (via shared.command_help) for anything unrecognised. Requires
  `manager` (the local QemuManager) to be importable — if not (server-only /
  orchestrator-only checkout) it prints a message directing the user to AI
  chat, since commands execute remotely via API_URL in that mode.

FUNCTION: tf_report(vm_name)
  IN:  vm_name str
  OUT: nothing (console output)
  Calls execute_tool("fingerprint_vm", {"name": vm_name}) and prints the
  report/error.

KEY HELPERS
  _show_stealth_popup(vm_name, setup_cmd) — shows the one-time stealth
    guest-setup instructions via a GUI popup: ctypes MessageBoxW on Windows,
    zenity then notify-send on Linux/macOS, tkinter as universal fallback.
    Silently gives up if no GUI toolkit is available.

IMPORTANT NOTES
  - "serve" sub-command starts `uvicorn orchestrator.http.api_server:app` on
    this machine — this is how you launch the orchestrator's client-facing
    HTTP API from the direct CLI instead of invoking uvicorn manually.
    (Previously broken: it imported a nonexistent `_EX` name from
    executor_client.py and pointed at the dead `client.server.api_server`
    module path — both fixed.)
  - "fetch" downloads a VM disk from a configured remote executor: fetches
    /images/{vm}/sha256 first, then streams /images/{vm} with HTTP Range
    resume support, then verifies the SHA-256 before declaring success.

orchestrator/ai/chat_turn.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The five interactive gates a single model tool call passes through before
  it executes, plus _process_tool_call() which drives one tool_call dict
  through all of them end to end: safety confirmation, pre-flight, the
  context assistant, the manual-config prompt, and clarify draining. The
  gate-independent base (TurnState, GateOutcome, and the pure arg transforms)
  lives in chat_types.py and is imported/re-exported here. Imports are
  one-directional (cli -> chat_turn -> chat_types) — no import cycle.

FUNCTION: _safety_gate(tool_name, raw_args, state, messages) -> GateOutcome
  IN:  tool_name str, raw_args dict, state TurnState, messages list[dict]
  OUT: GateOutcome — EXIT / CANCELLED / PROCEED
  Interactive confirmation before a mutating tool runs. delete_vm (and other
  _CRITICAL_TOOLS) double-confirm: type YES, then the exact VM name. Tools in
  _CONFIRM_YN confirm once with y/n and are batch-auto-confirmed for the rest
  of the turn once one instance of that tool type is approved. Everything
  else in _CONFIRM_NAME requires an exact-name retype. Skipped entirely when
  the value was already clarified/confirmed earlier in the same turn.

FUNCTION: _preflight_gate(tool_name, raw_args, state, messages, verbose)
          -> (raw_args, GateOutcome)
  IN:  tool_name str, raw_args dict, state TurnState, messages list[dict],
       verbose bool
  OUT: (dict, GateOutcome) — EXIT / REPLAN (abort) / CANCELLED / PROCEED
  Calls preflight._preflight_check() and acts on the result: "abort" appends
  an _INTERNAL_ nudge and returns REPLAN; "auto_fix" swaps in the corrected
  args; "ask_user" (for tools not already in _CONFIRM_NAME) shows the warning
  panel and prompts for a choice, patching fix_field or setting
  create_profile's force=True on approval.

FUNCTION: _context_assistant_gate(tool_name, raw_args, user_input,
                                  recent_context, state, messages)
          -> (raw_args, GateOutcome)
  IN:  tool_name str, raw_args dict, user_input str, recent_context str,
       state TurnState, messages list[dict]
  OUT: (dict, GateOutcome) — EXIT / REPLAN / PROCEED
  Fires context_assistant.check_context() once per turn. A "never mentioned
  it" hallucinated-field hint asks the user directly for that field and
  patches raw_args (PROCEED); a tool-mismatch/high-stakes hint pops the bad
  assistant message and nudges the AI to re-plan (REPLAN).

FUNCTION: _manual_config_gate(tool_name, raw_args, pre_gate_result, state)
          -> (raw_args, pre_gate_result, GateOutcome)
  IN:  tool_name str, raw_args dict, pre_gate_result dict|None, state TurnState
  OUT: (dict, dict|None, GateOutcome) — EXIT / PROCEED
  When create_vm was called with manual=True, prompts interactively for
  os/cpu/mem/disk overrides, applies simple regex-parsed values, marks
  os_type clarified, and drops any pending pre-gate clarify result (manual
  config owns the missing fields now).

FUNCTION: _clarify_drain(result, tool_name, state, messages) -> GateOutcome
  IN:  result dict (a clarify response), tool_name str, state TurnState,
       messages list[dict]
  OUT: GateOutcome — EXIT / SKIP_TOOL
  Drains every missing field from a clarify response in one pass (no
  per-field Ollama round-trip): prompts for each, handles the "overwrite"
  shortcut for name conflicts, and injects one _INTERNAL_ re-plan message
  summarising all the answers so the AI retries with exact values.

FUNCTION: _process_tool_call(tc, user_input, ui, state, messages, verbose)
          -> GateOutcome
  IN:  tc dict (one tool_call from the model), user_input str, ui str
       (lowercased), state TurnState, messages list[dict], verbose bool
  OUT: GateOutcome — PROCEED / EXIT / REPLAN / CANCELLED
  The orchestration function: parses the tool_call, checks for custom-mode
  activation, runs the context-assistant gate, resolves os_type, runs the
  preflight gate, runs the safety gate, builds the pre-execution gate result
  (missing gate-required fields not grounded in what the user actually said),
  runs the manual-config gate, then calls executor_client.execute_tool() (or
  uses the pre-gate clarify result instead), renders VNC/self-rendering
  tool output specially, appends the tool-result message, and drains any
  clarify response from the executor.

KEY HELPERS (imported from chat_types.py, re-exported here)
  TurnState, GateOutcome, _is_critical, _build_vm_spec_rows — see chat_types.py.

orchestrator/ai/chat_types.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The gate-independent foundation of the chat-turn pipeline: the TurnState
  dataclass, the GateOutcome enum, the critical-tool check, the create_vm
  spec-preview row builder, and the pure arg transforms (custom-mode
  detection, os_type resolution, pre-gate clarify construction). chat_turn.py
  and cli.py both import from here; this module imports nothing from either,
  so the edge is strictly one-directional.

CLASS: TurnState (dataclass)
  Mutable per-user-turn state passed through the gates instead of a dozen
  loose locals: user_wants_action, tools_called, tool_executed, last_had_tools,
  context_assistant_fired, clarified_fields/values, confirmed_values/
  tool_types (persist across the whole turn), and op_cancelled/
  clarify_happened/clarify_answer/clarify_field (reset every agentic round via
  reset_iteration()).

CLASS: GateOutcome (Enum)
  PROCEED / SKIP_TOOL / REPLAN / CANCELLED / EXIT — what a gate tells the
  tool loop to do next, replacing scattered break/continue/return so each
  gate stage can be extracted into its own function.

FUNCTION: _is_critical(tool_name, args) -> bool
  IN:  tool_name str, args dict
  OUT: bool — True only for tools in the config-driven critical_tools set
       (e.g. delete_vm) that require double confirmation.

FUNCTION: _build_vm_spec_rows(args) -> list[tuple]
  IN:  args dict (raw create_vm args)
  OUT: List[(label, value)] rows previewing what create_vm will actually use
  Mirrors the defaults tool_executor applies (profile matching, disk bus/
  format fallback, UEFI-for-Windows, hardened-forces-q35) so the preview
  shown before the y/n safety confirm matches the real outcome.

FUNCTION: _maybe_enable_custom_mode(tool_name, user_input_lower, messages)
  IN:  tool_name str, user_input_lower str, messages list[dict]
  OUT: nothing (side effect: calls preflight.set_custom_mode(True))
  For create_profile calls, scans the current + recent user messages for the
  word "custom" and disables product-name verification if found.

FUNCTION: _resolve_os_type(tool_name, raw_args, user_input_lower, state) -> dict
  IN:  tool_name str, raw_args dict, user_input_lower str, state TurnState
  OUT: dict (possibly-modified raw_args)
  For create_vm: if the user actually typed an OS keyword, pins os_type to
  its canonical alias and marks it clarified; if not, strips any os_type the
  AI guessed so the gate can ask instead of silently trusting a hallucination.

FUNCTION: _build_pre_gate_result(tool_name, raw_args, user_input,
                                 recent_context, state) -> dict | None
  IN:  tool_name str, raw_args dict, user_input str, recent_context str,
       state TurnState
  OUT: dict (clarify response) or None
  Checks gate-required fields against extract_slots(user_input) — not the
  AI's args — so a hallucinated value can't bypass the gate. A value the AI
  supplied is accepted only if it's grounded in the recent conversation text.

orchestrator/ai/http_chat.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  HTTP-mode single-turn chat processor. process_message() runs one user
  message through the same agentic tool loop as chat_loop() but with no
  stdin/stdout — it returns a dict instead of printing, so the orchestrator's
  /chat endpoint (orchestrator/http/api_server.py) can drive multi-turn
  conversations over HTTP with confirmation round-trips. Loads its own _CFG
  and imports every dependency from its own source module — it never imports
  from cli.py (one-directional edge: cli -> http_chat).

FUNCTION: process_message(user_input, messages, verbose=False,
                          auto_confirm=False) -> dict
  IN:  user_input str, messages list[dict], verbose bool, auto_confirm bool
  OUT: {"text": str, "messages": list, "tool_results": list,
        "needs_input": dict|None, "pending_tool": dict (only when
        needs_input is set)}
  Mirrors chat_loop's per-turn logic without any console I/O: runs the
  agentic loop (bounded by _LOOP_MAX), the context assistant, the os_type
  guard, preflight, the safety-confirmation gate, and the pre-execution gate
  — but instead of prompting interactively, any confirmation/clarification
  need is returned as a structured needs_input dict (type: confirm_yn /
  confirm_name / confirm_critical / preflight / clarify) plus a pending_tool
  the caller must resend with auto_confirm=True (and the exact typed name for
  confirm_critical) once the user answers.

IMPORTANT NOTES
  - This is the module orchestrator/http/api_server.py's POST /chat endpoint
    calls; its "fast path" (auto_confirm + a stored pending_tool) is handled
    directly in api_server.py rather than by re-entering process_message().

orchestrator/ai/ollama_client.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  HTTP client for the Ollama API. Builds the DOORMAN system prompt (tool-use
  rules, defaults, ISO two-step workflow, delete-vs-kill distinction, etc.)
  and sends conversation history + the TOOLS list to the model.

MODULE-LEVEL STATE
  OLLAMA_URL   — from env or config (default "http://localhost:11434")
  OLLAMA_MODEL — from env or config (default "qwen2.5:7b")

FUNCTION: _build_system_prompt() -> str
  IN:  nothing
  OUT: str — the full system prompt
  Pulls live OVMF availability and the profile-name list from
  executor_client, and the custom-mode flag from
  orchestrator.preflight.host_probe._CUSTOM_MODE, and interpolates them into
  a fixed prompt template covering tool-call rules (ACT vs ASK, defaults,
  name/machine-type/CPU/ISO rules, delete-vs-kill, case-insensitive VM name
  matching, user-rejection handling).

FUNCTION: _call_ollama(messages) -> dict
  IN:  messages List[dict]
  OUT: dict — the parsed Ollama /api/chat response ({"message": {...}})
  POSTs {model, messages: [system_prompt]+messages, tools: TOOLS, stream:
  false, options: {temperature, num_ctx}} to {OLLAMA_URL}/api/chat. On
  requests.ConnectionError: prints a friendly "start ollama serve / pull
  model" message and calls sys.exit(1).

orchestrator/ai/session.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Manages the AI conversation history (session). Loads/saves to
  ~/.qemu_vms/.session.json, filters out unverified assistant claims before
  persisting, and detects drift (a sign the model has been prompt-poisoned or
  is hallucinating success without ever calling a tool).

FUNCTION: load_session() -> List[dict]
  IN:  nothing
  OUT: list[dict] (messages), capped to the last MAX_SESSION_HISTORY entries.
  Returns [] if the file is missing or corrupt.

FUNCTION: save_session(messages)
  IN:  messages List[dict]
  OUT: nothing
  Filters the raw message list down to verified content only: drops tool
  messages, drops _INTERNAL_-prefixed content, keeps user messages, and keeps
  assistant text ONLY if it immediately followed a real tool result (a
  text-only "Done — X is created" with no preceding tool call is unverified
  and never persisted). Also trims trailing orphaned user messages. Writes
  the last MAX_SESSION_HISTORY entries to disk.

FUNCTION: detect_drift(messages) -> (str, str) | None
  IN:  messages List[dict]
  OUT: None (healthy), or ("critical"|"warn", message str)
  Two independent signals, each with a critical and a warn threshold (from
  config's drift_thresholds): (1) the ratio of user turns to verified
  assistant responses, and (2) the longest run of consecutive unanswered user
  turns. "critical" means the model is very likely poisoned and the session
  should be auto-cleared; "warn" is an early signal.

FUNCTION: clear_session()
  IN:  nothing → OUT: nothing — deletes the session file if present.

FUNCTION: set_auto_clear(enabled) / get by reading AUTO_CLEAR_SESSION
FUNCTION: set_loop_max(limit) -> int / get_loop_max() -> int
  Persist (to config.json) and read back the auto_clear flag and the
  tool_loop_max_override (None clears the override, falling back to the
  configured default).

orchestrator/ai/tools.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Loads the TOOLS list — the complete set of tool definitions passed to
  Ollama on every chat call — from orchestrator/ai/tools.json. The Python
  file itself is a 7-line loader; all the actual tool metadata (41 tool
  entries as of this writing: create_vm, launch_vm, stop_vm, list_vms,
  vm_status, monitor_vm, show_config, update_config, resize_disk, delete_vm,
  clone_vm, get_vm_logs, print_command, send_monitor_cmd, scan_isos,
  snapshot_create/list/restore/delete, create_profile/delete_profile/
  list_profiles/check_profile_compatibility, create_network/delete_network/
  list_networks/add_vm_to_network, set_resource_limits, open_display,
  open_shell, check_system, fingerprint_vm, add_label/remove_label/
  list_labels, mark_as_template/remove_template/list_templates, clarify,
  revert) lives in the JSON data file.

orchestrator/ai/tools.json
  Each entry is {"type": "function", "function": {"name", "description",
  "parameters": <JSON Schema>}} — the exact shape the Ollama /api/chat "tools"
  field expects. Editing tool descriptions or parameter schemas (to change
  what the model is told a tool does, or its required/optional args) means
  editing this file, not Python code.

orchestrator/ai/context_assistant.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Post-inject AI auditor. Runs after the model produces a tool call, before
  execute_tool fires. Checks whether the chosen tool matches what the prompt
  actually asked for, and whether the model invented a value for a field the
  user never mentioned. Returns a short _INTERNAL_ hint string for the model
  to self-correct on, or None if nothing looks wrong. Never blocks execution
  directly — it is a soft nudge; the context gate and preflight remain the
  hard stops.

FUNCTION: scan_tool_hints(prompt) -> List[str]
  IN:  prompt str
  OUT: list of tool names the prompt's trigger words hint at
  Applies specificity rules (e.g. "delete snapshot" suppresses the more
  generic delete_vm hint) so genuinely conflicting intent is the only thing
  that yields multiple hints.

FUNCTION: extract_slots(prompt) -> Dict[str, Optional[str]]
  IN:  prompt str
  OUT: dict of field -> literal value found in the prompt, or None if absent
  Regex/keyword scan against config-driven slot_patterns. Never infers —
  a bare "vm <word>" / "on <word>" pattern is only trusted if the captured
  token looks like a real identifier (contains a digit/dash/underscore), to
  avoid an ever-growing English-word denylist.

FUNCTION: check_context(prompt, tool_name, args, recent_context="",
                        known_names=None) -> Optional[str]
  IN:  prompt str, tool_name str, args dict, recent_context str (last N real
       user messages joined), known_names Optional[Set[str]] (ground-truth
       VM names, if the caller has them)
  OUT: None or a "|"-joined hint string
  Five checks, in order: (4 first) contradictory intent — the same name near
  triggers for two opposing tools (e.g. create AND delete) in one prompt;
  (1) tool mismatch against scan_tool_hints(); (2) hallucinated required
  fields — a required, trackable field the model set but the user never
  said; (3) high-stakes optional fields (dangerous booleans) set without user
  signal; (5) unknown VM reference — a name outside known_names for a
  VM-referencing tool. Recon tools (list_vms, scan_isos, check_system,
  list_profiles, list_networks, monitor_all/monitor_vm) are always exempt.

KEY HELPERS
  _strip_stopwords(text) — drops corporate filler words before product-string
                            token matching.
  _name_near_triggers(prompt, name, triggers) — True if name appears within
                            the configured word-proximity window of a trigger
                            phrase (used for the contradiction check).
  _trigger_in(text, trigger) — word-boundary match that also excludes
                            hyphen/underscore neighbours, so a VM name like
                            "build-box" doesn't false-positive on trigger
                            word "build".

orchestrator/event_log.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Persistent structured event log for every tool call routed through
  executor_client.execute_tool(). Every call is recorded as one JSON line.
  Used by the admin TUI's event feed and queryable via GET /events on the
  orchestrator's HTTP API.

LOG FILE
  ~/.qemu_vms/events.log   — active log
  ~/.qemu_vms/events.log.1 — previous log (after 10 MB rotation)

LOG FORMAT (one JSON object per line)
  {"ts": ISO-8601 UTC timestamp, "tool": tool name, "args": sanitised args
   dict (only name/src/dst/display/size_gb/tag/network/profile kept),
   "outcome": "ok" | "already_running" | error message, "duration_ms": float}

FUNCTION: log_event(tool, args, result, duration_ms)
  IN:  tool str, args Dict[str, Any], result Any, duration_ms float
  OUT: nothing
  Derives the outcome string from the result (via _summarise_result),
  strips args down to the allowed display keys (via _summarise_args), rotates
  the log file at 10 MB, and appends one JSON line. Opens the file in append
  mode each call — no held file handle. Never raises (wrapped in a bare
  try/except so a logging failure can't break the caller).

FUNCTION: read_events(limit=100, since="") -> list
  IN:  limit int, since str (ISO timestamp)
  OUT: list[dict], newest last (up to limit entries)
  Reads the current log file in reverse, optionally stopping at entries at
  or before `since`.

KEY HELPERS
  _summarise_args(tool, args)   — keeps only the display-safe keys.
  _summarise_result(result)     — "ok" / "already_running" / the error string.

orchestrator/executor_client.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  THE SEAM. The single point where the orchestrator decides: call the local
  executor in-process, or make an HTTP call to a remote executor.server
  instance. Everything above it is orchestrator/AI code; everything below it
  is executor code. The AI layer and the CLI import execute_tool only from
  here — never from shared.executioner.tool_executor or orchestrator.pipeline
  directly.

MODULE-LEVEL STATE (initialised at import time from connection_config.json)
  API_URL   — os.environ.get("API_URL", cfg["url"]) — "local" or a remote URL
  _TOKEN    — os.environ.get("EXECUTOR_TOKEN", cfg["token"]) — deliberately
              NOT API_TOKEN (that env var is the orchestrator's own
              client-facing secret used by orchestrator/http/api_server.py;
              reusing it here would leak the wrong secret to the executor)
  _TIMEOUT, _CA_CERT, _VERIFY — HTTP call tuning

  IN:  nothing
  OUT: dict {"profiles", "profiles_list", "ovmf", "capabilities"} (also
       stored in the module-level _synced cache)

  IN:  nothing each
  OUT: list / dict / dict / dict respectively
  Each returns the sync cache when populated; falls back to a direct

FUNCTION: check_profile_compatibility(profile_name) -> dict
  IN:  profile_name str
  OUT: {"compatible": bool, "issues": [...], "warnings": [...]}
  Delegates to execute_tool("check_profile_compatibility", ...) so the
  orchestrator never imports executor code directly for this check.

FUNCTION: execute_tool(tool_name, args, verbose=False, log=True) -> dict
  IN:  tool_name str, args dict, verbose bool, log bool (False for
       high-frequency internal polling that shouldn't drown the event log)
  OUT: dict, always containing "success": bool
  1. Enforces the tool allowlist.
  2. For launch_vm: overrides display to "vnc" when it's sdl/gtk/unset
     (nothing but a remote executor's own console can show sdl/gtk).
  3. Enforces the VM allowlist (reports "not found" rather than "forbidden"
     to avoid leaking existence) and the profile allowlist.
  4. Dispatches: local mode imports orchestrator.pipeline.execute_tool();
     remote mode POSTs {tool_name, args, verbose} to {API_URL}/execute with
     a Bearer token.
  5. Logs the call via orchestrator.event_log.log_event() unless log=False.
  6. In remote mode, patches open_display results so "localhost" becomes the
     real executor hostname.
  7. Filters list_vms results by the VM allowlist.

KEY HELPERS
  __getattr__(name) — lazily resolves "_execute_tool" to
  orchestrator.pipeline.execute_tool so mock.patch targets always hit the
  current binding rather than a stale import-time reference.

IMPORTANT NOTES
  - _VM_TOOLS is the set of tool names that reference an existing VM by name
    (used by the context assistant's unknown-VM-reference check and by
    _ALLOWED_VMS enforcement here).

orchestrator/http/api_server.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The orchestrator's FastAPI HTTP service. Exposes /chat (the AI loop),
  /execute (direct tool call), /health, /info, /events, /sync, /sessions,
  /rotate-token, /custom-mode, and the ship-image endpoints (/images/{vm},
  /images/{vm}/sha256, /vms/{vm}/bundle). Every route except /health requires
  a Bearer token — localhost callers are always trusted without one.

  Start with: uvicorn orchestrator.http.api_server:app --host 0.0.0.0 --port 8080

MODULE-LEVEL STATE
  _CFG (from orchestrator/connection_config.json), _ALLOWED_TOOLS,
  _LOCAL_ONLY_DISPLAYS, _MIN_TOKEN_LEN, _ALLOWED_VMS, _ALLOWED_PROFILES,
  _MAX_MESSAGE_LEN, _MAX_SESSIONS
  _TOKEN_FILE = ~/.gorgon.token ; _TOKEN loaded env-var-first, then file
  _sessions — in-memory dict keyed by session_id: {"messages", "pending_tool",
              "critical_step2", "last_active"}
  _SESSION_TTL_SECONDS — inactive sessions are evicted past this age

DEPENDENCY: _require_token(request, creds)
  Allows any request from 127.0.0.1/::1/localhost unconditionally. Otherwise
  re-reads the token fresh (so rotation takes effect without a restart) and
  does a constant-time compare (secrets.compare_digest) against the Bearer
  credential — raises HTTPException(401) on mismatch or if no token is
  configured server-side.

ENDPOINT: GET /health — no auth. {"status": "ok"}.
ENDPOINT: GET /info (auth) — Ollama model/URL + OVMF availability for the
  client banner.
ENDPOINT: GET /events (auth) — {"events": read_events(limit, since)}.
ENDPOINT: GET /sync (auth) — shortcut_commands, allowed_remote_tools, the
  authored command catalog (executor.command_catalog.COMMAND_CATALOG), the
  filtered VM list (name + status), and the filtered profile-name list — the
  client's one-shot startup sync.
ENDPOINT: POST /chat (auth) — the full multi-turn AI chat endpoint. Session
  state lives server-side in _sessions, keyed by session_id. Fast-path: if
  auto_confirm=True and a pending_tool is stored, skips Ollama entirely and
  either re-prompts for the second half of a critical (delete_vm-class)
  double-confirm, or executes the pending tool directly (injecting
  delete_disks=True for delete_vm) and appends a synthetic assistant/tool
  message pair to the session so the next Ollama call sees a completed
  action instead of repeating it. Otherwise calls
  orchestrator.ai.cli.process_message() (imported at call time from
  http_chat via cli's re-export) and stores the resulting messages/
  pending_tool.
ENDPOINT: GET /sessions (auth) / DELETE /sessions/{id} (auth) — list / clear
  in-memory chat sessions.
ENDPOINT: POST /rotate-token (auth) — validates length >= _MIN_TOKEN_LEN,
  updates the in-memory token, os.environ["API_TOKEN"], and writes
  ~/.gorgon.token with 0600 permissions from creation (opened O_CREAT|O_TRUNC
  with mode 0o600 up front, not chmod'd after the fact, to avoid a
  world-readable window).
ENDPOINT: POST /custom-mode (auth) — process-global toggle for
  orchestrator.preflight.validator.set_custom_mode(), matching the -cu flag's
  local-mode behaviour.
ENDPOINT: POST /execute (auth) — dispatch a single tool call: runs the
  server-side authoritative preflight (via a _manager_proxy() wrapper — see
  below), handles abort/auto_fix/ask_user the same way /chat's fast path
  does but as a single-shot HTTP response, forces vnc_bind_local=True and a
  vnc display override, and returns {"ok": True, "result": ...} (or raises
  HTTP 500 on an unhandled exception).
ENDPOINT: GET /images/{vm_name}/sha256, GET /images/{vm_name},
  GET /vms/{vm_name}/bundle (all auth) — SHA-256 checksum, ranged qcow2
  disk stream, and tar.gz VM-folder bundle respectively. In local mode these
  read the disk directly off ~/.qemu_vms/; in split/remote mode they proxy
  the equivalent request to the configured executor's own /vms/{vm}/disk*
  endpoints (see executor/server.py) and stream the response straight
  through.

HELPER: _manager_proxy() -> object
  Returns a QemuManager wrapper in local mode, or a thin executor_client
  proxy (scan_isos/list_vms only) in remote mode. Both branches filter
  list_vms() by _ALLOWED_VMS — without this, preflight's own VM-existence
  checks would see a hidden VM as real and skip its "not found" handling, a
  side channel that would leak the hidden VM's existence even though the
  tool call itself would still be denied.

HELPER: _filter_allowed(names, allowlist) -> list
  Returns names as-is if allowlist is empty (unrestricted); else the
  intersection.

IMPORTANT NOTES
  - This file replaces the old server/http/api_server.py — the module path is
    now orchestrator.http.api_server, and start-server code across the repo
    (admin_tui.py, chat_client.py's autostart, direct_cli.py's "serve") must
    reference "orchestrator.http.api_server:app".

orchestrator/preflight/validator.py

PURPOSE
  The pre-flight gate: cross-checks sanitised tool-call args against real
  system/VM state (or shape-only checks in stateless mode) and returns a
  structured action dict the caller must act on before dispatch.
  _preflight_check and _show_preflight_warning are the two entry points
  chat_turn.py, http_chat.py, and orchestrator/http/api_server.py call.
  Imports the internet/host-probing half of the old monolithic validator
  from host_probe.py (one-directional: validator -> host_probe).

MODULE-LEVEL STATE
  _CFG / _THRESHOLDS — from orchestrator/preflight/config.json
  _QEMU_CPU_MODELS, _LAPTOP_TYPE_KEYWORDS, _PREFLIGHT_TOOLS,
  _PREFLIGHT_HW_FIELDS, _DESTRUCTIVE_MON_CMDS, _BAD_ISO_PATH_PATTERNS

FUNCTION: _preflight_check(tool_name, args, manager, verbose=False,
                           stateless_only=False) -> dict
  IN:  tool_name str, args dict, manager object (QemuManager or a proxy;
       passed in to avoid a circular import), verbose bool, stateless_only
       bool (True on the AI provider in remote mode — skip checks needing
       real filesystem/binary/manager state; the executor machine always
       runs the full check)
  OUT: {"action": "ok"|"auto_fix"|"ask_user"|"abort", ...}
  Returns "ok" immediately for any tool not in _PREFLIGHT_TOOLS. Dispatches by
  tool_name: create_vm to _preflight_create_vm(); create_profile checks it
  has at least one hardware field and, unless force=True, runs
  _validate_profile_for_host + _validate_with_internet; launch_vm checks VM
  existence (with a "did you mean" suggestion) and warns if the attached ISO
  file is missing; delete_vm/remove_template/snapshot_restore/
  snapshot_delete ask for confirmation unless force=True; resize_disk checks
  existence and blocks shrinking; send_monitor_cmd asks for confirmation on
  destructive HMP commands.

FUNCTION: _preflight_create_vm(args, manager, verbose, stateless_only) -> dict
  IN:  args dict, manager object, verbose bool, stateless_only bool
  OUT: action dict (see above)
  The largest single check: missing/placeholder name -> ask_user; existing
  name -> ask_user (overwrite/rename); the destructive unattended-install
  confirmation (WIPES the disk — separate wording for Windows account
  auto-creation vs. Linux stopping at account creation); invalid
  machine_type used as a profile name -> auto_fix; bad/missing ISO path ->
  auto_fix (resolved) or ask_user (pick from scan results); ARM64 ISO vs
  x86_64 VM -> ask_user; Windows without UEFI -> auto_fix; Windows disk below
  the minimum -> auto_fix; Windows without swtpm installed -> ask_user;
  stealth-args validation via _validate_stealth_args() (auto-fixes inferred
  SMBIOS fields, blocks on hard "error" issues, or non-blocking on
  "warning"); internet/QEMU validation via host_probe._validate_with_internet();
  profile-vs-host validation via host_probe._validate_profile_for_host().

FUNCTION: _validate_stealth_args(args) -> List[dict]
  IN:  args dict
  OUT: list of issue dicts (severity: "error"/"warning"/"auto_fix")
  Checks manufacturer/product_name/smbios_type/bios_vendor/bios_version/
  serial_number presence (auto-inferring from product_name via
  host_probe._stealth_infer_from_product where possible), GPU choice (warns
  on "virtio", errors on SPICE display), uefi=False (errors — SeaBIOS is a
  clear tell), and a QEMU-synthetic cpu_model (errors).

FUNCTION: _triage(issues) -> (blockers, auto_fixes, warnings)
  IN:  issues List[dict]
  OUT: three lists partitioned by severity/auto_fix.

FUNCTION: set_custom_mode(val) — re-exported from host_probe; sets the
  global that disables product-name internet verification (-cu flag).

FUNCTION: _show_preflight_warning(preflight, console)
  IN:  preflight dict, console object
  OUT: nothing (console output) — a yellow "Pre-flight Check" panel with the
  reason/correction/question/options.

PURPOSE

  Six checks: (1) machine_type supported by the installed QEMU; (2) CPU model
  recognised; (3) CPU/arch consistency (ARM CPU + x86_64 VM or vice versa,

  identity fields.

  Keyword-matches product_name against a configured hint table (e.g.
  "thinkpad" -> Lenovo/Notebook).

orchestrator/sanitizer/sanitizer.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Transforms raw AI-generated tool args into clean, safe args. Never blocks,
  never raises — always returns a dict. This is the very first step of
  orchestrator.pipeline.execute_tool().

MODULE-LEVEL STATE
  PLACEHOLDER_VM_NAMES, PLACEHOLDER_ISO_PATTERNS — config-driven placeholder
  detection sets
  VALID_MACHINE_TYPES / _DISPLAY_MODES / _GPU_TYPES / _AUDIO_TYPES /
  _NETWORK_MODES / _DISK_FORMATS / _BIOS / _MACHINE_ARCH / _MACHINE_CLASS /
  _OS_TYPES — enum validation sets
  OS_TYPE_ALIASES — os_type normalisation map
  REAL_HOME — os.path.expanduser("~"), used to fix wrong-username paths

FUNCTION: _sanitise_args(tool_name, args) -> dict
  IN:  tool_name str, args dict
  OUT: a NEW dict (the caller's dict is never mutated)
  Steps: coerce int/bool fields (stripping "GB"/"mb" suffixes); strip special
  chars from and reject placeholder VM names; reject profile names used as
  machine_type (auto-mapping to the profile field instead); apply OS_TYPE_
  ALIASES; validate every enum field against its valid set (falling back to
  a configured default); fix path fields via _fix_path(); force kvm=False +
  hugepages=False for ARM/raspi configs; reject an ARM cpu_model on an x86_64
  VM; validate or drop a malformed MAC address; reject a raw ethernet/wifi
  bridge_iface (falls back to the default bridge); cap memory_mb/cpu_cores/
  cpu_threads/disk_size_gb to configured bounds (memory further capped to a
  fraction of live host RAM via psutil); validate port fields; sanitise
  extra_args to a list of short strings; strip non-alphanumerics from
  snapshot/network/profile names; strip path-like characters from text
  fields; drop empty optional fields entirely.

FUNCTION: _resolve_vm_name(vms, ref) -> Optional[str]
  IN:  vms List[dict], ref str
  OUT: resolved VM name, "all" (pass-through), or None
  Tries exact match -> 1-based index match ("2" -> second VM) -> substring
  match against name or OS.

FUNCTION: _resolve_iso(iso_hint) -> Optional[str]
  IN:  iso_hint str (a path or a bare distro name)
  OUT: resolved absolute path, or the original hint if nothing was found
  Resolution order: exact path -> wrong-username-fixed path -> fuzzy keyword
  score across all search directories (OS-keyword expansion plus a 10-point
  bonus for a direct basename substring match) -> first ISO found anywhere.

KEY HELPERS
  _fix_path(p) — rejects literal placeholder path patterns, fixes
                 /home/<wrong-user>/ and Windows C:\ style paths to the real
                 home directory.

executor/server.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The executor's own standalone FastAPI HTTP service. Receives already-
  validated tool calls from the orchestrator over HTTP and executes them
  against the local QEMU engine directly via
  shared.executioner.tool_executor.dispatch_tool() — no AI, no NLU, no

  call reached here). This is what makes the executor able to run as its

  Start with: uvicorn executor.server:app --host 0.0.0.0 --port 8001

  Environment variables:
    EXECUTOR_TOKEN  shared secret between orchestrator and executor — can
                    also be written to ~/.gorgon-executor.token or set as
                    "token" in executor/config.json.

MODULE-LEVEL STATE
  _CFG (executor/config.json), _TOKEN_FILE = ~/.gorgon-executor.token,
  _EVENT_LOG = ~/.gorgon-executor-events.jsonl (separate from the
  orchestrator's own event log — meaningful when the two run on different
  machines), _KNOWN_TOOLS (the full set of dispatchable tool names,
  hand-mirrored from tool_executor.py's dispatch table)

DEPENDENCY: _require_token(request, creds)
  Same shape as the orchestrator's own — trusts localhost unconditionally,
  else requires an exact (non-constant-time — unlike api_server.py's
  secrets.compare_digest) match against the loaded executor token.

ENDPOINT: GET /health — no auth. {"status": "ok", "version": "1.0"}.
ENDPOINT: GET /profiles (auth) — {"profiles", "profiles_list", "ovmf"} — what
  orchestrator.executor_client.sync() calls in remote mode.
ENDPOINT: GET /capabilities (auth) — check_system_capabilities() dict.
ENDPOINT: GET /tools (auth) — {"tools": sorted(_KNOWN_TOOLS)} — lets the
  orchestrator verify on startup that this executor supports every tool it
  intends to dispatch.
ENDPOINT: GET /commands (auth) — {"commands", "category_order"} straight
  from executor.command_catalog — the single source of truth both help
  surfaces render from.
ENDPOINT: GET /status (auth) — {"status": "ok", "vms_total", "vms_running"}
  or {"status": "degraded", "error"} on failure.
ENDPOINT: POST /execute (auth) — {"tool_name", "args", "verbose"} -> runs
  dispatch_tool(), logs locally via _log_event(), and ALWAYS returns a JSON
  body (never a bare HTTP 500) so the orchestrator can parse the error even

ENDPOINT: GET /vms/{vm_name}/disk (auth) — streams the primary qcow2 disk
  with SHA-256 header + HTTP Range support (206 partial / 200 full).
ENDPOINT: GET /vms/{vm_name}/disk/sha256 (auth) — {"vm_name", "disk",

ENDPOINT: GET /vms/{vm_name}/bundle (auth) — streams the entire VM directory

HELPER: _log_event(tool_name, args, result, duration_ms)
  Appends one JSON line to _EVENT_LOG — separate from and simpler than

  different machine than the orchestrator).

HELPER: _disk_path(vm_name) -> pathlib.Path
  Raises HTTPException(404) if the VM directory or a qcow2 file inside it is
  missing; otherwise returns the first (sorted) *.qcow2 match.

  Checks per section: inxi -M (manufacturer/product/BIOS vendor+version/
  serial/chassis type), inxi -N (MAC OUI prefix, NIC driver model), inxi -C
  (hypervisor CPUID flag, CPU model string), inxi -D (disk bus, disk model
  string), inxi -A (audio chip), inxi -G (GPU driver), plus a "host security"
  section reporting whether hardened mode's seccomp sandbox/hidden-hypervisor-
  bit/mitigation-flags/SMM-off are active.

IMPORTANT NOTES

executor/api/_vm_lifecycle.py  (_VmLifecycleMixin)
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  VM creation, cloning, golden-image templates, and deletion — the layer
  that has grown the most since the two-tier layout: it now handles
  template-based cloning (backing-file disk clones from a marked golden
  image) and wires in the offline credential/hostname randomization modules
  for each fresh clone.

FUNCTION: scan_isos() -> List[Dict]
  IN:  nothing → OUT: [{"name", "path", "size_gb"}, ...]
  Scans every directory from qemu_arg_builder.build_iso_search_dirs() for
  *.iso files, de-duplicated by full path.

FUNCTION: create_vm(config: MachineConfig, force=False) -> dict
  IN:  config MachineConfig (name must be set), force bool (deprecated —
       ignored; callers must pre-delete)
  OUT: {"success": True, "name", "vm_dir", "bios", "uefi", "iso_path",
       "iso_name", "os_name", "message", ...optional randomization fields}
       or {"success": False, "error"}
  Creates vm_dir, applies apply_os_hints(), finds/copies the right
  OVMF_VARS.fd variant (searching _4M variants first for modern distros,
  falling back through a long candidate list including macOS/Windows QEMU
  install paths; falls back to SeaBIOS with a printed warning if nothing is
  found), auto-assigns VNC/SPICE ports. For each disk: if config.template is
  set, creates a qemu-img backing-file clone off the template's flattened
  disk (config.template must have been produced by mark_as_template()
  first); otherwise a fresh blank qcow2. When cloning from a template AND
  the corresponding randomize_* flags are set, offline-edits the freshly
  cloned disk BEFORE the VM ever boots: randomize_root_password /
  new_username / randomize_user_password (Linux templates only, via
  ._vm_credentials), and randomize_hostname (both Linux and Windows
  templates, via ._vm_hostname) — each is best-effort, recording a *_warning
  string in the result message on failure rather than blocking VM creation.
  Auto-attaches a matching ISO via _match_iso() if none was given AND this
  isn't a template clone (a clone already has a real bootable OS — attaching
  an installer ISO risks booting the installer menu instead). Infers
  os_name from the ISO filename if not given. Saves the config.

FUNCTION: clone_vm(source_name, new_name) -> dict
  IN:  source_name str (must be stopped), new_name str
  OUT: {"success": True, "message", "new_vm"} or error dict
  Makes an independent (non-backing) qemu-img clone of each disk, copies
  OVMF_VARS.fd, assigns a fresh vm_id and MAC per network interface, saves
  the new config under new_name.

FUNCTION: mark_as_template(name) -> dict
  IN:  name str (must be stopped)
  OUT: {"success": True, "message", "template"} or error dict
  Flattens each disk (qemu-img convert, NOT a backing-file link — so the
  template never depends on the source VM's own disk surviving) into
  ~/.qemu_vms/_templates/<name>/diskN.qcow2, writes template.json, and tags
  the source VM with the protected "template" label (registered via
  label_registry.register_label()).

FUNCTION: remove_template(name) -> dict
  IN:  name str → OUT: {"success": True, "message"} or error dict
  Deletes the template directory and un-tags the source VM's "template"
  label if the source VM still exists. Gated behind a Yes/Cancel
  confirmation at the preflight layer (see "remove_template" in
  orchestrator/preflight/validator.py's _PREFLIGHT_TOOLS); this method itself
  deletes unconditionally once called, same as delete_vm.

FUNCTION: list_templates() -> List[Dict]
  IN:  nothing → OUT: [{"name", "os_type", "disks" (count)}, ...]

FUNCTION: delete_vm(name, delete_disks=False) -> dict
  IN:  name str (must be stopped), delete_disks bool
  OUT: {"success": True, "message"} (message notes any disk-removal
       warnings) or error dict.

KEY HELPERS
  _match_iso(os_type, os_name, machine_arch) -> List[Dict] — scores every
    scanned ISO against os_type/os_name keywords (10x bonus for a specific
    os_name word match over a generic os_type keyword), filtering out
    architecture-mismatched ISOs entirely.
  _template_dir(name) — path helper for ~/.qemu_vms/_templates/<name>.

executor/api/_vm_runtime.py  (_VmRuntimeMixin)
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  VM launch and stop, swtpm lifecycle, and unattended-install completion
  detection. (Note: this is the mixin that owns launch/stop now — creation/
  cloning/templates live in _vm_lifecycle.py; the names swapped meaning
  relative to the pre-restructure two-tier layout.)

FUNCTION: launch_vm(name, display=None, dry_run=False, vnc_bind_local=None)
          -> dict
  IN:  name str, display Optional[str] (per-launch override), dry_run bool,
       vnc_bind_local Optional[bool] (per-launch override; the orchestrator
       sets this explicitly for remote/split-mode launches)
  OUT: {"success": True, "pid", "display", ...} — VNC launches also include
       "vnc_port" and, when vnc_bind_local, "vnc_password"; dry_run returns
       {"success": True, "dry_run": True, "command": str} without executing.
       {"success": False, "already_running": True, ...} if already running.
       {"success": False, "error"} on any failure.
  Loads the config, starts swtpm first if tpm is set, calls
  _maybe_finish_unattended_install() to detect and clear a completed
  install before rebuilding args, builds the QEMU command via
  QemuArgBuilder, Popen()s it (start_new_session on POSIX, a new process
  group on Windows), waits a brief liveness window and reports failure with
  the launch.log tail if QEMU exited immediately, records the PID, applies
  CPU pinning if configured. For unattended Windows installs on first boot:
  spawns a background thread that spams "sendkey ret" over the HMP monitor
  socket for ~25s to clear both the "press any key to boot from CD" prompt
  AND Windows Setup's language-selection screen (a single fixed window tuned
  empirically — see the long code comment for why an adaptive stop was tried
  and reverted). For VNC with vnc_bind_local: generates an exactly-8-character
  password (VNC auth is DES/8-byte — anything longer would silently truncate
  and desync the displayed vs. effective password), sets and 4-hour-expires
  it via QMP. Generates and serves the stealth guest-setup script on first
  launch if stealth and not yet done. If an ISO is attached: writes a
  ".relaunch_after_install" flag and spawns a detached watcher subprocess
  that waits for this QEMU process to exit, then (if the flag still exists)
  relaunches the VM automatically — used for Windows installs that reboot
  mid-setup.

FUNCTION: stop_vm(name, force=False) -> dict
  IN:  name str, force bool (skip graceful shutdown, SIGKILL immediately)
  OUT: {"success": True, "name", "message"} or error dict
  Cancels any pending auto-relaunch flag first (an intentional stop must
  win). Unless force: tries QMP system_powerdown and waits up to a
  configured grace period. Then terminate()/kill() the tracked process
  handle. Always reaps the VM's swtpm daemon via _stop_swtpm().

FUNCTION: stop_all() -> dict
  IN:  nothing → OUT: {name: stop_vm() result, ...} for every tracked VM.

FUNCTION: _maybe_finish_unattended_install(config) -> bool
  IN:  config MachineConfig
  OUT: bool — True if installer fields (iso_path, kernel_path, initrd_path,
       kernel_cmdline) were cleared and the config persisted
  Two-stage, cheapest-first check: (1) qemu-img info actual-size >= 2GB
  (skip the expensive stage for an obviously-still-blank disk); (2) a real
  read-only check for /etc/os-release via virt-cat
  (_vm_credentials.linux_os_installed) — disk-size alone can false-positive
  mid-partitioning, before the OS is actually installed. Exists specifically
  to prevent a real recurring bug: without clearing the kernel-boot fields,
  every future relaunch (including a guest-initiated post-install reboot)
  re-triggers the SAME destructive auto-partitioning preseed and wipes the
  just-completed install.

KEY HELPERS
  _start_swtpm(config) / _stop_swtpm(name) — start the swtpm socket daemon
    (with --terminate so it self-exits when QEMU disconnects, since Ubuntu's
    AppArmor profile blocks this framework's own SIGTERM/SIGKILL) and reap it
    by pidfile plus a psutil sweep for orphaned instances bound to the same
    tpm dir (relaunches/crashes can leave the pidfile stale).
  _find_qemu_pid(name) — locates an orphaned QEMU process by
    "process=<name>" in its cmdline (survives session restarts).
  _start_setup_server(name, script_path) -> int — one-off HTTP server on a
    random port serving the stealth guest-setup script; returns the port.

executor/api/_vm_credentials.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Offline per-clone Linux credential randomization. Golden-image cloning
  backing-clones a template's disk byte-for-byte, so every clone would
  otherwise inherit the exact same /etc/shadow content. This edits the
  cloned disk directly via virt-customize (libguestfs) — no boot required,
  so it runs safely during create_vm before the VM is ever started. Linux
  only: Windows credentials live in the SAM registry hive and need a
  different mechanism (see _vm_hostname.py's Windows path for the same
  guestfish/hivexregedit pattern applied to hostnames instead).

FUNCTION: virt_customize_available() -> bool
  Whether virt-customize (libguestfs-tools) is on PATH.

FUNCTION: linux_os_installed(disk_path) -> bool
  IN:  disk_path str
  OUT: bool — True only if /etc/os-release exists and is non-empty via
  virt-cat (read-only, offline). Used by _vm_runtime._maybe_finish_
  unattended_install to distinguish "genuinely installed" from "merely
  partitioned so far" (a raw disk-size check can false-positive on the
  latter).

FUNCTION: randomize_root_password(disk_path) -> str
  IN:  disk_path str (must not be attached to a running VM)
  OUT: the generated 16-char random password
  Raises RuntimeError if virt-customize is missing or the operation fails
  (e.g. no recognisable Linux root filesystem on the disk).

FUNCTION: rename_user(disk_path, new_username, old_username=None) -> str
  IN:  disk_path str, new_username str (must match POSIX username rules —
       also closes off shell injection into the --run-command string),
       old_username Optional[str] (auto-detected via find_primary_user() if
       omitted)
  OUT: new_username (returned for convenience/logging)
  Runs the guest's own usermod -l -m -d (moves the home dir too) plus
  groupmod, correctly handling the Debian/Kali/Ubuntu private-group-per-user
  convention. Raises RuntimeError on any validation or virt-customize
  failure.

FUNCTION: find_primary_user(disk_path) -> Optional[str]
  IN:  disk_path str
  OUT: the first non-system account (UID >= 1000, excluding "nobody") from
  /etc/passwd via virt-cat, or None.

FUNCTION: randomize_user_password(disk_path, username) -> str
  IN:  disk_path str, username str
  OUT: the generated 16-char random password (via virt-customize --password)
  Raises RuntimeError on failure (e.g. the username doesn't exist on disk).

executor/api/_vm_hostname.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Offline per-clone hostname/computer-name randomization, mirroring
  _vm_credentials.py's mechanism. A golden-image clone otherwise inherits
  the exact same OS-level hostname — a real cross-clone fingerprint. Linux
  uses virt-customize's built-in --hostname. Windows has no equivalent tool,
  so this shells out to guestfish + hivexregedit directly: extract the
  SYSTEM registry hive, patch the ComputerName/Tcpip Hostname values with
  hivexregedit, write the hive back.

FUNCTION: virt_customize_available() -> bool
FUNCTION: windows_hostname_tools_available() -> bool
  Whether the required tool(s) are on PATH (virt-customize; guestfish +
  hivexregedit respectively).

FUNCTION: generate_linux_hostname(disk_path) -> str
  IN:  disk_path str → OUT: a new hostname keeping the current distro-flavored
  prefix (read via virt-cat) but replacing any generic "-vm"/"-template"
  suffix with a random 7-char one; falls back to "linux-vm" if unreadable.

FUNCTION: generate_windows_hostname() -> str
  IN:  nothing → OUT: a "DESKTOP-XXXXXXX" name matching Windows' own
  auto-generated-name convention.

FUNCTION: randomize_linux_hostname(disk_path, hostname=None) -> str
  IN:  disk_path str, hostname Optional[str] (auto-generated if omitted)
  OUT: the hostname that was set — via virt-customize --hostname (updates
  /etc/hostname and /etc/hosts). Raises RuntimeError on failure.

FUNCTION: randomize_windows_hostname(disk_path, hostname=None) -> str
  IN:  disk_path str, hostname Optional[str]
  OUT: the hostname that was set
  Extracts SYSTEM via guestfish (read-only mount of the fixed Windows
  partition path), writes a .reg file setting ComputerName/
  ActiveComputerName/Tcpip Hostname + "NV Hostname" as UTF-16LE hex values,
  merges it with hivexregedit, runs a best-effort ntfsfix (a normal Windows
  shutdown typically leaves NTFS "dirty", which blocks a read-write mount
  even with remove_hiberfile — ntfsfix clears the journal as a side effect
  even though its own hibernation check separately errors), then writes the
  patched hive back to the disk with a remove_hiberfile,rw mount (needed
  because a freshly-installed Windows disk is normally still marked
  hibernated by Fast Startup, which otherwise forces a read-only NTFS
  mount). Raises RuntimeError on any step's failure.

executor/api/_qemu_smbios.py  (_QemuSmbiosMixin)
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  SMBIOS/chassis identity arg-building mixin, composed into QemuArgBuilder.
  Operates purely on builder state (self.cfg / self.args / self.vm_dir); split
  out to keep qemu_arg_builder.py focused on the rest of the command line.

FUNCTION: _smbios()
  IN:  nothing → OUT: appends args to self.args
  Emits -smbios type=1 (manufacturer/product/serial/family — family is
  derived from the product name's first token, NEVER the hostname, to avoid
  leaking "localhost" into dmidecode/inxi), type=0 (BIOS vendor/version),
  type=2 (baseboard manufacturer/product — overrides the "QEMU"/"Standard PC"
  defaults inxi reads via DMI to identify KVM), and type=3 (chassis — via a
  raw injected binary since QEMU 8.x has no CLI way to set the chassis_type
  byte; falls back to a plain -smbios type=3,manufacturer= if the binary
  write fails).

FUNCTION: _chassis_type_byte() -> int
  IN:  nothing → OUT: the SMBIOS chassis-type byte for the config's
  smbios_type (falling back to machine_class if smbios_type is unset),
  mapped via config.json's smbios_chassis_type_map.

FUNCTION: _write_smbios_chassis_bin(chassis_type) -> str
  IN:  chassis_type int → OUT: path to the written smbios_chassis.bin, or ''
  on failure/ARM/chassis_type==0. Writes a raw SMBIOS type=3 structure that
  QEMU appends after its own built-in structures — Linux's DMI scanner
  overwrites dmi_chassis_type on every type=3 hit, so the appended entry
  (ours) wins over QEMU's built-in default (Other).

FUNCTION: _smbios_escape(value) -> str (static)
  IN:  value str → OUT: value with commas removed — a comma inside a
  -smbios value would terminate the field and start a new key=value pair,
  which is an injection vector into arbitrary QEMU SMBIOS directives.

executor/api/qemu_arg_builder.py  (QemuArgBuilder)
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Translates a MachineConfig into the full QEMU command-line argument list.
  Composed with _QemuSmbiosMixin (see above). Also still owns/re-exports the
  QEMU-version and port-pool/ISO-search-dir helpers via
  executor/api/qemu_host_utils.py.

FUNCTION: build() -> List[str]
  IN:  nothing → OUT: the complete argv, starting with the QEMU binary name,
  empty strings stripped
  Calls, in order: _base, (_harden if hardened), _machine, _cpu, _memory,
  (_firmware unless raspi), _smbios, _disks, _network, _display, _audio,
  (_usb unless raspi), _battery, _kernel_direct, _qmp, _monitor, _serial,
  (_misc unless arm), (_tpm if configured and not arm). Filters any
  extra_arg that disables the seccomp sandbox when hardened is set — a last
  line of defense after the sanitiser, since this is the final step before
  the command is assembled.

KEY SUB-METHODS
  _base()      — -name, -enable-kvm (unless ARM — KVM never works for ARM
                 guests on an x86 host).
  _harden()    — CPU masking (-hypervisor, kvm=off, -vmx, +invtsc — the
                 +invtsc addition defeats the naive "no invariant TSC" VM
                 tell without touching VMEXIT-latency red-pills), disables
                 balloon/hugepages, forces non-stealth networks to NAT, and
                 adds the QEMU seccomp -sandbox flag.
  _machine()   — -machine (with accel=kvm, kernel_irqchip=on, and ACPI OEM
                 ID/table-ID overrides derived from manufacturer/
                 product_name), -rtc.
  _cpu()       — -cpu (preset lookup + feature flags), -smp topology.
  _memory()    — -m, optional hugepages -mem-path, virtio-balloon device.
  _firmware()  — OVMF code+vars pflash drives (x86 only).
  _disks()     — SCSI/SATA controllers as needed, per-disk args via
                 DiskConfig.to_qemu_args(), the install ISO cdrom, and (when
                 unattended=True) the generated answer-CD/cidata volume.
  _network()   — per-NetworkConfig args, or a default user-NAT NIC if none
                 configured.
  _gl_available() — probes whether virgl/OpenGL actually works before
                 passing gl=on (avoids a broken display).
  _display()   — GPU passthrough (vfio-pci — the only way to defeat a
                 "VMware SVGA" style tell, hands the guest a REAL GPU's PCI
                 IDs) or an emulated GPU device (stealth picks vmware-svga
                 on Linux / plain VGA on Windows; non-stealth uses
                 cirrus-vga to avoid Bochs VGA's QEMU-identifying PCI ID),
                 plus -display sdl/gtk/spice/vnc/-nographic.
  _audio()     — detects PulseAudio/pipewire/coreaudio/dsound and emits the
                 matching -audiodev + device, skipping entirely if no audio
                 server is running (avoids a crash).
  _usb()       — NEC xHCI controller (real chip PCI ID, unlike qemu-xhci's
                 Red Hat/QEMU ID) + keyboard + mouse/tablet (stealth forces
                 usb-mouse — an absolute-positioning tablet is itself a VM
                 tell), plus the unattended-Windows FAT answer-media stick.
  _battery()   — synthetic ACPI battery/AC adapter for laptop personas (via
                 a precompiled acpi/battery.aml SSDT, inert if absent).
  _kernel_direct() — -kernel/-initrd/-append for unattended Linux installs.
  _qmp() / _monitor() / _serial() — Unix sockets on Linux/macOS, TCP on
                 Windows.
  _tpm()       — TPM 2.0 emulator chardev/tpmdev/device.
  _misc()      — virtio-rng (non-hardened only), -no-reboot when an ISO is
                 attached, -no-user-config.

CLASS: DiskConfig / NetworkConfig / MachineConfig — see qemu_config.py below
  (this module imports them, does not define them).

PURPOSE

    username="", password="", locale="", organization=None,

PURPOSE

LAYOUT
  Row 0       Header: gorgon + spinner while waiting + server URL + VM list
  Row 1       Separator
  Rows 2..h-5 Scrollable chat history — last N lines of _history
  Row h-4     Separator
  Row h-3     "> " + input buffer  (or "waiting for response..." spinner)
  Row h-2     Hint line (list / system / profiles / templates / drift /

THREADING MODEL
  Main thread: curses draw loop (100ms timeout) + keyboard input.
  Worker thread (daemon, one per in-flight request): POSTs to /chat and puts
  the JSON result into _resp_q. The main loop drains _resp_q each tick, sets
  _waiting=False, and calls _process_response().

MODULE-LEVEL STATE
  _history, _lock, _resp_q, _waiting, _needs_confirm, _is_confirm,
  _pending_kill, _session_id (persisted to ~/.qemu_vms/.chat_session_id),
  _REMOTE_VMS/_REMOTE_PROFILES/_COMMANDS/_ALLOWED_TOOLS (from /sync),
  _SC_LIST/_SC_SYSTEM/_SC_PROFILES/_SC_TEMPLATES/_SC_DRIFT/_SC_CLEAR/
  _SC_HELP/_EXIT_CMDS (shortcut trigger sets, overridable by /sync)

FUNCTION: chat_loop(verbose=False, color_hex="#aaaaaa", font_size=13)
  IN:  verbose bool, color_hex str, font_size int → OUT: does not return
  (blocks until quit) — wraps _run() in curses.wrapper().

FUNCTION: _run(stdscr, verbose, color_hex, font_size)
  Initialises curses/colors, resizes the terminal via an xterm escape
  sequence, loads/creates the persisted session id, and — if
  _is_localhost() and the server isn't already reachable — calls
  _autostart_server(stdscr) before proceeding. Then _sync_from_server(),
  prints the connection/VM/profile summary, and runs the main input loop:
  drains _resp_q, redraws, reads one key at a time, and on Enter either
  handles a pending force-kill y/n confirmation, dispatches a built-in
  shortcut via _dispatch(), or spawns an _http_worker thread to send the
  message to /chat.

FUNCTION: _autostart_server(stdscr) -> bool
  IN:  stdscr → OUT: bool — True once the server responds to /health
  Locates orchestrator/http/api_server.py relative to this file (client_dir
  -> files_dir -> orchestrator/http/api_server.py) and, if present, launches
  `python -m uvicorn orchestrator.http.api_server:app` as a detached
  subprocess (PYTHONPATH set to files_dir, API_TOKEN read from
  ~/.gorgon.token if present), polling /health for up to
  _AUTOSTART_POLLS * _AUTOSTART_INTERVAL_S seconds. Logs to
  /tmp/gorgon-orchestrator.log (configurable). Returns False (with a
  "Could not start server" message) if the module isn't found or never
  becomes reachable.

FUNCTION: _sync_from_server() -> bool

FUNCTION: _dispatch(cmd, verbose) -> bool
  IN:  cmd str, verbose bool → OUT: bool — True if handled locally (no HTTP
  needed). Handles: exit/quit shortcuts, "clear session"/"/clear" (DELETEs
  the server-side session and starts a fresh session_id), help, list [label],
  system, profiles, templates, drift, and "kill <vm>"/"force stop <vm>"/
  "force kill <vm>"/"hard stop <vm>" (sets _pending_kill and asks for y/n

FUNCTION: _process_response(result, verbose=False)
  IN:  result dict (a /chat response), verbose bool → OUT: nothing

  and sets _needs_confirm/_is_confirm from the response's needs_input.

FUNCTION: _render_tool_result(tool, result)
  IN:  tool str, result dict → OUT: nothing (appends formatted lines to
  the scrollback) — per-tool compact rendering for list_vms, launch_vm
  (auto-opens a VNC viewer via _try_open_vnc()), check_system, create_vm
  (with an ISO/declared-OS mismatch warning via _iso_distro_hint()),

shared/executioner/tool_executor.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  The executor-side tool dispatch layer. Owns the process-wide QemuManager
  singleton and the _run() dispatch function that every tool call ultimately
  reaches. Has NO orchestrator imports — it runs standalone on an
  executor-only machine. Two entry points sit on top of _run(): dispatch_tool
  (used by executor/server.py — pre-validated args, no gate/preflight) and
  orchestrator.pipeline.execute_tool (which supplies the real
  sanitiser/gate/preflight callables for local-mode single-machine use).

MODULE-LEVEL STATE
  _CFG/_VM_DEFS/_TOOL_DEFS/_VALID_MACHINE_TYPES/_ARM_CPU_PREFIXES/
  _GENERIC_OS_NAMES/_ISO_ARM_KEYWORDS/_ISO_X86_KEYWORDS — from
  shared/executioner/config.json
  manager — the single QemuManager() instance, constructed at import time
  _last_revert_action — {"tool","args","description"} of the last reversible
    op, or {}
  _REVERT_AWARE_TOOLS — tools that manage _last_revert_action themselves
    (set it on success, or explicitly clear it), excluded from the blanket
    "any other tool clears revert" rule so an unrelated failed call doesn't
    wipe a still-valid revert.
  _IDENTITY_STOPWORDS — corporate filler words ("inc", "corp", "llc", ...)
    excluded from product-name profile matching so every "…Inc." request
    doesn't collide with every "…Inc." profile.

FUNCTION: dispatch_tool(tool_name, args, verbose=False) -> Any
  IN:  tool_name str, args dict (already sanitised/gated/resolved by the
       orchestrator), verbose bool
  OUT: tool result — calls _run() with stub sanitiser/preflight/
       resolve_iso callables (the orchestrator already ran the real ones).
       This is what executor/server.py's POST /execute calls.

FUNCTION: _execute_create_vm(args, verbose, raw_os_type, placeholder_vm_names,
    resolve_iso) -> dict
  IN:  args dict, verbose bool, raw_os_type str (captured before the
       sanitiser may alias os_type), placeholder_vm_names set, resolve_iso
       Callable
  OUT: create_vm result dict
  Extracted from _run() to keep the dispatch table readable — the single
  largest tool handler. Validates/sanitises the name, handles overwrite
  (deletes the existing VM first), matches a library profile by a genuine
  product-name token (never a manufacturer/stopword token — the old logic's
  "any >3-char token" match let a MacBook borrow a Dell profile via "inc."),
  auto-picks a random stealth persona when stealth is set with no explicit
  identity (stealth_persona._pick_stealth_persona — form-factor pinnable via
  a stealth_persona arg), applies the profile, copies the remaining args onto
  the config, registers user labels, applies GPU-passthrough/unattended-
  install opt-ins, enforces stealth-implies-hardened plus the per-model
  BIOS/RAM/CPU variance (stealth_persona._apply_within_model_variance) and a
  unique per-unit serial/BIOS-vendor/BIOS-version fallback, forces TPM on for
  Windows 11 unless explicitly disabled, forces q35+no-balloon+no-hugepages
  when hardened, rejects an invalid machine_type or an ARM CPU on an x86_64
  VM, detects and enforces ARM64-ISO-implies-aarch64-VM (with an explicit
  arch-mismatch abort otherwise), assembles the disk (SATA for stealth/
  Windows — a spoofable real SSD model string vs. NVMe's fixed "QEMU NVMe
  Ctrl" tell or virtio-blk's /dev/vd* + Red Hat PCI ID) and network configs
  (including the stealth slirp_subnet rewrite), auto-resolves an ISO from a
  distro-name hint via resolve_iso() when none was given, and finally calls
  manager.create_vm(cfg). On success: generates the unattended-install media
  (Windows autounattend.iso via executor.api.autoinstall.windows, or Linux
  kernel/initrd+cidata/preseed via executor.api.autoinstall.linux, keyed off
  the distro's installer_family) and sets the delete_vm revert action.

FUNCTION: _run(tool_name, args, verbose, *, raw_os_type="",
    placeholder_vm_names=None, resolve_iso=None, preflight_check=None,
    show_preflight_warning=None) -> Any
  IN:  tool_name str, args dict (pre-pipeline — all orchestrator-side
       concerns must already be done), verbose bool, plus the injected
       callables (defaulted to no-op stubs when called via dispatch_tool)
  OUT: Any — the raw tool result (list/dict, per tool)
  The full tool-name dispatch table: revert (re-runs the stored inverse
  action after an interactive y/n confirm — refuses if stdin isn't a TTY),
  clarify, check_system, scan_isos, list_vms, add_label/remove_label/
  list_labels, mark_as_template/remove_template/list_templates,
  list_profiles, check_profile_compatibility, create_profile (runs its own
  preflight unless force=True, records a delete_profile revert),
  delete_profile, create_vm (delegates to _execute_create_vm), clone_vm,
  launch_vm, stop_vm, vm_status, monitor_vm, show_config, update_config
  (captures old values first so the revert can restore them), resize_disk,
  snapshot_create/list/restore/delete, set_resource_limits, create_network/
  delete_network/list_networks/add_vm_to_network, open_display, open_shell,
  delete_vm, check_disk, get_vm_logs, print_command, fingerprint_vm,
  send_monitor_cmd. Calls the matching shared.display render_*() function
  for each tool's result unless verbose=True.

FUNCTION: _set_revert(tool, args, description) / _clear_revert()
  Set/clear _last_revert_action — called by every revert-aware tool handler
  on success/irreversibility respectively.

KEY HELPERS
  See shared/executioner/stealth_persona.py for the extracted stealth
  fingerprint-data generators this module imports (_plausible_bios_version,
  _apply_within_model_variance, _generate_disk_model, _pick_stealth_persona,

shared/executioner/stealth_persona.py
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Pure helpers that manufacture believable hardware-identity values for a
  stealth VM: a plausible BIOS version, per-unit RAM/CPU variance within a
  model's real option set, a real SATA SSD disk-model string, a random
  realistic hardware persona pick, and a vendor-plausible serial number.
  Kept apart from tool_executor.py because they are self-contained and drive
  off a private RNG (_STEALTH_RNG) so create_vm never perturbs the global
  `random` stream other callers/tests depend on. This module imports nothing
  from tool_executor.py — the edge is one-directional (tool_executor imports
  these back).

FUNCTION: _plausible_bios_version(manufacturer) -> str
  IN:  manufacturer str → OUT: a "1.N.N" version string — used when a
  user-named model isn't in the profile library, so a coherent-looking
  firmware version can be synthesised rather than borrowed from an unrelated
  profile.

FUNCTION: _apply_within_model_variance(cfg, profile_data, args)
  IN:  cfg MachineConfig (mutated in place), profile_data dict (may carry
       bios_versions/memory_options_mb/cpu_variants), args dict (an explicit
       caller value is never overridden)
  OUT: nothing
  Picks BIOS version, RAM, and CPU core/thread count from the applied
  profile's own curated real-option lists (never invented) so two stealth
  VMs of the same model aren't byte-identical beyond their already-unique
  serial and MAC. RAM is capped near 50% of live host RAM (via psutil);
  cpu_variants are [cores, total_threads] and get converted to threads-per-
  core, capped to the host's CPU count.

FUNCTION: _generate_disk_model() -> str
  IN:  nothing → OUT: one of a curated list of real SATA SSD model strings
  (Samsung 870 EVO, Crucial MX500, WD, Kingston, SanDisk, SK hynix, Intel,
  Micron, Toshiba), so lsblk/inxi -D/smartctl show believable hardware.

FUNCTION: _pick_stealth_persona(form_factor="", os_type="") -> str
  IN:  form_factor str ("laptop"/"desktop"/"server", or ""/"any"/"random"
       for no constraint), os_type str (guest OS — excludes personas pinned
       to a different OS, e.g. a Mac mini persona under a Linux guest)
  OUT: a profile name from get_all_profiles(), or "" if none qualify.
  Assigning a fresh random persona per VM is what lets rotation defeat
  long-term fingerprinting across many stealth VMs.

FUNCTION: _generate_stealth_serial(manufacturer) -> str
  IN:  manufacturer str → OUT: a serial string shaped like that vendor's
  real serial format (Dell 7-char alnum service tag, Lenovo 2-letter+6-alnum,
  HP 10-char digit/letter/alnum, Apple "C02"+8-alnum, Supermicro "S"+9-digit,
  else a generic 8-char alnum fallback). Unique per-unit serials are
  essential for rotation — two same-model VMs must never share one.

admin/  (package; entry point admin/admin_tui.py)
────────────────────────────────────────────────────────────────────────────────
PURPOSE
  Standalone curses fullscreen admin dashboard. Runs as its OWN process
  (`gorgon-admin`), completely independent of the orchestrator/executor —
  it can run on any machine that can reach the orchestrator over HTTP, and
  is configured entirely via admin/config/ (connection_config.json +
  admin_config.json, over admin_config.defaults.json). Once one admin_tui.py
  file; now an admin/ package (config, state, api_client, server_control,
  render, commands, keyboard, app) behind a thin admin_tui.py entry point that
  keeps the gorgon-admin alias path stable. Commands are one class per file
  under admin/commands/. This is a real behavior change from the
  pre-restructure two-tier layout: the admin TUI no longer calls
  manager.launch_vm()/manager.stop_vm() etc. in-process. Every VM operation
  (launch/stop/kill/stopall/list) now goes over HTTP through the
  orchestrator's real /execute and /events endpoints using a Bearer token
  (from API_TOKEN, or ~/.gorgon.token, or admin/config/connection_config.json's
  "token" key). Only start-server / shutdown / kill-server / status remain
  local-process operations — and even those only work meaningfully when the
  admin TUI happens to run on the same machine as the orchestrator (they
  pgrep for a local "api_server" process).

  Launch:  gorgon-admin  (alias added by the install scripts)
  Config:  admin/config/connection_config.json (orchestrator_url, token),
           admin/config/admin_config.json (text_color hex, font_size, refresh_rate_s,
           default_port, log_path, events_display_limit)

LAYOUT (curses, differential updates — no flicker)
  Row 0      Header bar: uptime, VM count, event count, orchestrator
             online/unreachable + URL
  Rows 1+    "VMs" section rule, column header, then the VM table (name,
             ●/○ status, CPU, RAM, OS) sized to available rows
  Rows +     "Recent Events" section rule, column header, then the event
             feed (time, tool, target, outcome, ms)
  Last 3     Separator + command prompt (with inline result message) + hint
             line

KEYBOARD HANDLING
  All curses calls (get_wch + erase/addstr/refresh) happen on the single
  main-loop thread — reading input on a separate thread while the loop draws
  previously caused real, confirmed screen corruption (curses/ncurses is not
  safe to call concurrently against the same window from multiple threads).
  _handle_key() applies each keypress synchronously; _dispatch() (which may
  block on a network call) still runs on its own daemon thread, but touches
  only shared Python state under _lock, never curses itself. Enter dispatches
  the command buffer to a background thread running _dispatch(). "help" sets
  _help_mode; any real keypress (not a raw curses navigation event) while
  help is open clears it back to the dashboard.

COMMANDS (entered at the command prompt)
  launch <vm> / start <vm>  — POST /execute {"launch_vm", {"name": vm}}
  stop <vm>                 — POST /execute {"stop_vm", {"name": vm,
                               "force": False}}
  kill <vm>                 — POST /execute {"stop_vm", {"name": vm,
                               "force": True}}
  stopall                   — lists VMs via /execute, stops every running one
  list                       — lists VM names via /execute into the prompt message
  start-server               — launches `uvicorn orchestrator.http.api_server:app`
                               as a detached local subprocess (only meaningful
                               if this admin TUI runs on the orchestrator's
                               own machine); polls _local_pid() afterward
  shutdown / shutdown-server — SIGTERM to the local orchestrator process
                               (found via `pgrep -f api_server`)
  kill-server                — SIGKILL to the local orchestrator process
  status                     — reports orchestrator online/unreachable + VM/
                               running counts (via /health and /execute)
  help                        — fullscreen help overlay (any real key dismisses)
  q / Esc / Ctrl-C            — quit

KEY HELPERS
  _post(path, body) / _get(path, params) — raw HTTP helpers with the Bearer
    token attached; return {} / an error-shaped dict on any failure rather
    than raising.
  _exec(tool_name, args=None, log=True) — POST /execute and unwrap the
    {"ok", "result"} envelope. Always passes verbose=True (this is a machine
    caller polling once a second — without it the orchestrator would print a
    full Rich table to its own console/log every tick). log=False is used
    only for the dashboard's own automatic background refresh, never for a
    command the operator typed, so routine polling doesn't drown the shared
    event feed.
  _get_events(limit=200) — GET /events.
  _server_online() — GET /health, 2-second result cache.
  _local_pid() — `pgrep -f api_server`, used only by the local-process
    commands.
  _allowed_tools() — mirrors client/cli/commands/context.py's helper exactly: reads
    orchestrator.executor_client._ALLOWED_TOOLS so the help overlay doesn't
    advertise commands that would actually be rejected (dispatch already
    enforces the allowlist for free via /execute -> execute_tool(); this is
    purely about honest help text).
  _draw_help(stdscr, h, w) — reuses the same curses window across frames
    (recreating it every tick was measured to generate ~27x the idle
    dashboard's terminal traffic for fully static content — the confirmed
    cause of a help-only flicker bug).

IMPORTANT NOTES
  - _ORCH_URL defaults to "http://localhost:8080" (env SERVER_URL, or
    connection_config.json's "orchestrator_url").
  - The token precedence is: API_TOKEN env var -> connection_config.json's
    "token" -> ~/.gorgon.token file.
  - Because every VM operation goes through /execute, the orchestrator's own
    tool allowlist, VM allowlist, and preflight checks all apply to the
    admin TUI exactly as they would to any other client — there is no
    privileged bypass path here.
