================================================================================
  GORGON  —  GENERAL WORKFLOW  (handbook part 2)
================================================================================

This document traces every workflow in the project from the user's perspective
all the way down to the system call. Understand this document and you understand
how every piece connects.

Three machine roles, three top-level packages: orchestrator/ (AI + gating +
the client-facing HTTP API), executor/ (owns QEMU/KVM directly, can run as its
own standalone HTTP service), client/ (thin remote UI). shared/ holds code used
by more than one of them. admin/ is a separate standalone dashboard, not part
of this request lifecycle.

Even in the common single-machine setup, the client is a real HTTP client of
the orchestrator — "gorgon" does NOT run an in-process chat loop by default.
The only in-process chat loop is orchestrator/ai/cli.py's chat_loop(), reached
by running cli.py directly as a script (orchestrator-side debugging), not by
the "gorgon" alias.

────────────────────────────────────────────────────────────────────────────────
1. STARTUP SEQUENCE
────────────────────────────────────────────────────────────────────────────────

When you run "gorgon" the following happens in order:

  1. Shell expands the alias (written by setup_client.sh / install_orchestrator.sh):
       alias gorgon='PYTHONPATH=~/Desktop/gorgon/files python3 ~/Desktop/gorgon/files/client/client_wrapper.py'

  2. Python executes client/client_wrapper.py [main()]:
     - Parses flags: -v (verbose), -cu (custom mode), -cs (clear session),
       -tf <vm> (fingerprint report — calls client/cli/commands/ directly,
       no AI, and returns immediately)
     - Loads CLI_config.json (accent colour, font size) from the client/ dir
     - If argv is non-empty (e.g. "gorgon list"): imports client/cli/commands/
       and calls run(argv, verbose) — a direct local QEMU command, no AI/HTTP
       involved for the VM operation itself (see section "DIRECT LOCAL COMMANDS"
       below)
     - If argv is empty (plain "gorgon"): imports client/ui/chat_client/ and
       calls chat_loop(verbose, color_hex, font_size) — the curses chat TUI

  3. chat_client.py's chat_loop() [client/ui/chat_client/ → _run()]:
     - Reads connection_config.json (server_url, token, timeout, verify_ssl)
       from the client/ directory; SERVER_URL/API_TOKEN env vars override it
     - Loads/generates a persistent session id from ~/.qemu_vms/.chat_session_id
     - Initialises curses colours from CLI_config.json's text_color
     - Prints "Connecting to {SERVER_URL}..."
     - Calls _is_localhost() — True only when SERVER_URL's hostname is
       localhost/127.0.0.1/::1
     - If localhost AND _server_reachable() (GET {SERVER_URL}/health) fails:
         _autostart_server() — spawns
           python3 -m uvicorn orchestrator.http.api_server:app --host 0.0.0.0
             --port <port from SERVER_URL>
         as a detached subprocess (PYTHONPATH set to the files/ dir, API_TOKEN
         read from ~/.gorgon.token if present), logs to /tmp/gorgon-orchestrator.log,
         then polls /health up to autostart_poll_count times (default 20 ×
         0.5s) waiting for it to come up
     - Calls _sync_from_server() — GET {SERVER_URL}/sync — pulls the shortcut
       command words, the VM list, the profile list, the command catalog, and
       the allowed-remote-tools set, used to populate the header bar and the
       in-TUI "help" command
     - Enters the curses main loop: draw, read a line of input, dispatch

  4. On the orchestrator side (whether autostarted or already running),
     api_server.py's @app.on_event("startup") handler calls
     orchestrator/executor_client.py's sync() — pulls profiles/OVMF/capabilities
     either from a direct executor.api.qemu_config import (API_URL == "local")
     or via GET {API_URL}/profiles + /capabilities on the executor server
     (split mode).

Split-mode variant: if SERVER_URL points at a non-localhost machine (or an
SSH-tunnelled localhost port fronting a remote orchestrator), _is_localhost()
is False and the client never tries to autostart anything — it just POSTs to
whatever's on the other end of SERVER_URL and reports "Cannot connect" if
nothing answers.

Alternate entry point (bypasses the client and HTTP entirely): running
"python3 -m orchestrator.ai.cli" directly (no args) calls chat_loop() in
orchestrator/ai/cli.py — the interactive REPL that talks to Ollama and the
executor pipeline in the same process, with no client_wrapper.py, no curses
TUI, and no /chat HTTP hop. This is for orchestrator-side debugging only; the
"gorgon" alias never invokes it.

────────────────────────────────────────────────────────────────────────────────
2. THE FULL REQUEST LIFECYCLE (one user prompt, step by step)
────────────────────────────────────────────────────────────────────────────────

This traces a single user prompt all the way through to the QEMU command,
in the default single-machine setup (orchestrator and executor co-located,
connection_config.json "url": "local").
Example: User types "create a linux vm called dev-box with 4gb ram"

STEP 1 — INPUT CAPTURE  [client/ui/chat_client/ → _run() / _dispatch()]
  - The curses main loop reads a line of input from the terminal
  - _dispatch(cmd) checks it against built-in shortcuts first (list, system,
    profiles, templates, drift, clear session, help, exit/quit/q/bye, and the
    "kill <vm>" force-stop shortcut) — these run via a direct POST /execute
    call and never touch the AI
  - If not a shortcut: a background thread POSTs the message to {SERVER_URL}/chat
    with {"message", "session_id", "auto_confirm", "verbose"}; the TUI shows
    "waiting for response..." while the thread runs

STEP 2 — ORCHESTRATOR RECEIVES THE CHAT REQUEST  [orchestrator/http/api_server.py → chat()]
  - FastAPI's _require_token dependency allows localhost freely; otherwise
    requires a valid Bearer token (compared with secrets.compare_digest)
  - Looks up (or creates) the in-memory session by session_id — history,
    any pending_tool from a previous confirmation round, and critical_step2 state
  - If auto_confirm=True and a pending_tool is stored (the user just answered
    a confirmation prompt): takes the fast path — skips Ollama entirely, may
    require a second "type the VM name" step for critical tools (delete_vm),
    then calls execute_tool() directly and returns
  - Otherwise (normal path): calls orchestrator/ai/http_chat.py's
    process_message(user_input, messages, verbose, auto_confirm)

STEP 3 — ONE HTTP CHAT TURN  [orchestrator/ai/http_chat.py → process_message()]
  This is the HTTP-mode equivalent of the interactive REPL loop in cli.py —
  same gates, no stdin/stdout. It runs its own tool loop (up to config.json's
  chat.tool_loop_max iterations), calling _call_ollama() each round and, for
  every tool_call the model returns, running it through the gates below
  in order. When a gate needs user input it does NOT prompt a terminal — it
  returns {"needs_input": {...}} to api_server.py, which returns it to
  chat_client.py, which shows the question/options in the TUI and waits for
  the next message (sent with auto_confirm=True if it was a yes/no confirm).

STEP 4 — OLLAMA CALL  [orchestrator/ai/ollama_client.py → _call_ollama()]
  - _build_system_prompt() assembles the system message: live VM/profile
    state, OVMF availability, the "DOORMAN" persona and ACT-vs-ASK rules,
    machine-type/CPU/ISO constraints, and (if -cu/custom mode is active) a
    note disabling product-name warnings
  - POSTs to {OLLAMA_URL}/api/chat with model=OLLAMA_MODEL (default qwen2.5:7b),
    messages=[system, ...history...], tools=TOOLS (from orchestrator/ai/tools.py),
    stream=false
  - Parses the response: tool_calls present → dispatch each below; otherwise
    it's a plain text reply (returned to the user, no tool executed this turn)

STEP 5 — PER-TOOL-CALL GATES  [orchestrator/ai/http_chat.py, inline]
  For each tool_call the model returned, in order:
  a) Context assistant check — check_context() (context_assistant.py) flags
     semantic mismatches (wrong VM named, a hallucinated required field never
     mentioned by the user). A hallucinated-field hint is turned into a
     needs_input clarify; a mismatch is fed back to the model as an
     "_INTERNAL_" nudge to re-plan, without executing anything.
  b) OS-type guard — if the user's own words contain an OS keyword
     (ubuntu/windows/mint/...), it overrides whatever os_type the model chose,
     via OS_TYPE_ALIASES (orchestrator/sanitizer/sanitizer.py).
  c) Preflight check — orchestrator/preflight/validator.py's _preflight_check()
     runs with the real manager (full state check, since this is local mode).
     abort → fed back to the model as an internal correction, no needs_input.
     auto_fix → args replaced silently.
     ask_user → returned as needs_input (type "preflight") and the tool_name/
       args are stashed in the session's pending_tool for the confirm round-trip.
  d) Safety confirmation — config-driven CONFIRM_YN / CONFIRM_NAME tables
     (config.json) decide whether the tool needs a yes/no or an exact-name
     confirmation; delete_vm additionally requires typing the VM name a
     second time (the critical_step2 flow in api_server.py). Skipped if this
     exact (field, value) was already confirmed/clarified earlier this turn.
  e) Pre-execution gate — orchestrator/sanitizer/context_gate.py's required-
     field table (_REQUIRED, loaded from context_gate_config.json) is checked
     inline against the args and the user's own recent words; anything still
     missing becomes a clarify response instead of executing.

STEP 6 — EXECUTE  [orchestrator/executor_client.py → execute_tool()]
  If none of the gates above short-circuited: process_message() calls
  execute_tool(tool_name, raw_args, verbose) — imported from
  orchestrator/executor_client.py, NOT called via orchestrator/pipeline.py
  directly. This function:
  - Enforces the tool/VM/profile allowlists from connection_config.json
    (empty lists mean "no restriction")
  - Forces launch_vm's display to "vnc" if it was sdl/gtk or unset (SDL/GTK
    can't render over HTTP to a remote client)
  - Times the call, then branches on API_URL:
      "local"           → imports and calls orchestrator/pipeline.py's
                          execute_tool() in-process (see STEP 7)
      "http://host:8001" → POSTs {"tool_name","args","verbose"} to
                          {API_URL}/execute on the executor server (split
                          mode — see the SPLIT MODE section below instead)
  - Logs the call via orchestrator/event_log.py's log_event() (unless log=False)
  - Filters list_vms results by the VM allowlist before returning

  Note: orchestrator/sanitizer/sanitizer.py's _sanitise_args() and
  orchestrator/sanitizer/context_gate.py's gate_check() (the full enum-
  normalisation / resource-cap / placeholder-stripping pass, distinct from the
  lighter inline checks in STEP 5) are called ONLY from inside
  orchestrator/pipeline.py — i.e. only on the local-mode path. See the SPLIT
  MODE section for what that means when the executor is on another machine.

STEP 7 — PIPELINE (local mode only)  [orchestrator/pipeline.py → execute_tool()]
  - _sanitise_args(tool_name, args) — the full sanitiser pass (section 7 below)
  - gate_check(tool_name, args) — every tool's required-field table is
    checked again here (belt-and-suspenders with STEP 5e); any miss returns
    a clarify dict straight back up through executor_client → http_chat
  - VM name resolution — for any tool taking an existing "name" arg (except
    create_vm/create_profile/clone_vm/create_network): manager.list_vms() is
    fetched and _resolve_vm_name() fuzzy-matches (exact / 1-based index /
    substring) the raw name to a real one
  - Dispatches to shared/executioner/tool_executor.py's _run(), passing in
    the real preflight_check/resolve_iso/placeholder_vm_names implementations
    (this is the same _run() the executor's standalone dispatch_tool() calls
    with no-op stubs instead — see SPLIT MODE)

STEP 8 — TOOL DISPATCH  [shared/executioner/tool_executor.py → _run()]
  The dispatch switch handles every tool name:
  - create_vm → _execute_create_vm() builds a MachineConfig (profile
    matching/application, stealth persona assignment, SMBIOS/disk/network
    assembly, ISO auto-resolution, arch consistency checks) then calls
    manager.create_vm()
  - launch_vm / stop_vm / vm_status / snapshot_* / clone_vm / delete_vm / etc.
    → thin wrappers around the corresponding QemuManager method
  - Also manages _last_revert_action (set on success of reversible ops, used
    by the "revert" tool) and renders Rich console output unless verbose=True

STEP 9 — QEMU MANAGER  [executor/api/qemu_manager.py, via the _Vm*Mixin classes]
  For launch_vm specifically [_vm_runtime.py → launch_vm()]:
  1. MachineConfig.load(name) reads config.json from the VM's directory
  2. Returns "already_running" immediately if a live process is tracked/found
  3. If config.tpm: _start_swtpm() launches the swtpm socket daemon and waits
     (up to 3s) for tpm.sock to appear
  4. QemuArgBuilder(config).build() [executor/api/qemu_arg_builder.py] builds
     the full argv list
  5. subprocess.Popen(cmd, ...) starts QEMU, detached (start_new_session=True);
     a brief liveness sleep (0.3s) catches immediate crashes (bad args, no
     /dev/kvm) before reporting success
  6. Records the PID in memory and in the VM's persisted state
  7. If config.cpu_pinning: taskset-pins the process after a short delay
  8. If display == "vnc": connects over QMP (up to 15 × 0.5s retries) and,
     when vnc_bind_local is set, calls QMP set_password with a random 8-char
     password and expire_password ~4 hours out
  9. If config.stealth and the one-time setup hasn't run yet: generates the
     guest_setup script and starts a small HTTP server thread to serve it
  10. If an ISO/kernel is attached: spawns a detached watcher subprocess that
      relaunches the VM once QEMU exits, for unattended-install flows

STEP 10 — ARG BUILDER  [executor/api/qemu_arg_builder.py → QemuArgBuilder.build()]
  Builds the full QEMU command-line argument list from MachineConfig:
  - -name, -machine, -cpu, -smp, -m (memory)
  - -drive (disk), -netdev/-device (network)
  - -display (sdl/gtk/vnc/none), -vga or -device (GPU)
  - -audiodev/-device (audio), -bios or -drive (firmware)
  - -rtc, -monitor, -serial, -qmp (sockets)
  - Stealth: -machine acpi_table=..., a real USB controller device, SMBIOS
    type=0/1/2 fields, CPUID/BIOS variance
  - Hardened: -cpu ...,kvm=off,-vmx
  - TPM: -chardev, -tpmdev, -device
  - extra_args appended last

STEP 11 — RESULT RETURN AND DISPLAY
  The result dict bubbles back up: tool_executor → pipeline → executor_client
  → http_chat's process_message() (appended to messages as a "tool" role
  entry) → api_server.py's chat() response body → chat_client.py's
  _process_response() in the TUI, which:
  - Renders per-tool formatted output via _render_tool_result() (VM tables,
    the check_system panel, launch_vm's VNC line — and auto-launches a local
    VNC viewer if one is installed)
  - Shows the model's short acknowledgement text, if any
  - If needs_input is present: shows the question/options and waits for the
    next message (sent with auto_confirm=True for a yes/no or critical confirm)
  Session history (messages, pending_tool) is kept server-side, keyed by
  session_id — the client only persists the session_id itself, in
  ~/.qemu_vms/.chat_session_id.

────────────────────────────────────────────────────────────────────────────────
2A. SPLIT MODE — WHEN THE EXECUTOR IS ON A SEPARATE MACHINE
────────────────────────────────────────────────────────────────────────────────

Steps 1–6 above (client → orchestrator, Ollama, the gates in http_chat.py) are
IDENTICAL whether or not the executor is co-located — the orchestrator always
runs the AI loop, the context assistant, the preflight check, and the safety/
gate checks itself, using its own manager import (or, in preflight, with
stateless_only=True and manager=None so it can't touch real filesystem state
it doesn't have).

The difference is entirely inside STEP 6's execute_tool()
[orchestrator/executor_client.py]: when connection_config.json's "url" (or
the API_URL env var) is not "local" but an executor URL like
"http://executor-host:8001", execute_tool() does NOT import
orchestrator/pipeline.py at all. Instead it POSTs
  {"tool_name": ..., "args": ..., "verbose": ...}
to {API_URL}/execute with an "Authorization: Bearer {EXECUTOR_TOKEN}" header
(EXECUTOR_TOKEN is a separate secret from the orchestrator's own client-facing
API_TOKEN — the two are never the same value).

On the executor machine, executor/server.py receives it:
  1. _require_token — localhost-exempt, otherwise Bearer-token check against
     EXECUTOR_TOKEN (env var → ~/.gorgon-executor.token → config.json's
     "token" field, in that precedence order)
  2. Validates tool_name against _KNOWN_TOOLS (a fixed set mirroring
     tool_executor.py's dispatch table)
  3. Calls shared/executioner/tool_executor.py's dispatch_tool(), which calls
     the SAME _run() dispatch function as the local-mode pipeline path, but
     wired with no-op stubs instead of real ones:
       preflight_check    → always returns {"action": "ok"} (a no-op)
       resolve_iso        → identity function (returns the path unchanged)
       placeholder_vm_names → empty set (nothing is treated as a placeholder)
  4. Logs the call to its own local ~/.gorgon-executor-events.jsonl (a
     separate log from the orchestrator's ~/.qemu_vms/events.log — the two
     machines don't share a filesystem)
  5. Returns the result dict, which flows back through executor_client.py to
     http_chat.py exactly like the local-mode result would

The key consequence: orchestrator/sanitizer/sanitizer.py's _sanitise_args()
and orchestrator/sanitizer/context_gate.py's gate_check() — the deep
arg-cleaning pass (enum normalisation, memory/CPU/disk caps, path fixing,
bridge/MAC rejection) — live ONLY in orchestrator/pipeline.py, which split
mode never calls. What actually protects a split deployment is: the
inline gates already run in http_chat.py/chat_turn.py before execute_tool()
is ever called (context assistant, OS-type guard, the full _preflight_check()
with stateless_only=True, the safety confirmation, and the inline
_GATE_REQUIRED check) — plus the executor's own preflight_check stub is
inert by design, since re-validating already-orchestrator-approved args a
second time on the executor would just duplicate work the orchestrator
already did.

executor/server.py also exposes GET /profiles, /capabilities, /status,
/tools, /commands, and the VM-disk/bundle streaming endpoints
(/vms/{name}/disk, /disk/sha256, /bundle) that orchestrator/http/api_server.py
proxies through for `gorgon fetch` / `gorgon bundle` in split mode.

────────────────────────────────────────────────────────────────────────────────
2B. THE INTERACTIVE REPL PATH (orchestrator/ai/cli.py chat_loop) — NOT the
    normal "gorgon" flow
────────────────────────────────────────────────────────────────────────────────

Running orchestrator/ai/cli.py directly (chat_loop(), no client, no HTTP)
follows almost the same gate sequence as http_chat.py, but the gates
themselves live in orchestrator/ai/chat_turn.py's _process_tool_call() and
prompt the terminal interactively (console.input()) instead of returning
needs_input over HTTP — e.g. _safety_gate() prints the confirmation and
blocks on stdin right there, and _preflight_gate() prints the Rich warning
panel and reads the user's choice directly. Both chat_turn.py and http_chat.py
ultimately call the same orchestrator/executor_client.py execute_tool(), so
STEPS 6 onward are identical either way — only how confirmations/clarifies
reach the user differs (terminal prompt vs. HTTP needs_input round-trip).

────────────────────────────────────────────────────────────────────────────────
2C. DIRECT LOCAL COMMANDS (gorgon list / launch / stop / ...) — NO AI
────────────────────────────────────────────────────────────────────────────────

When "gorgon <cmd> [args]" is run with a non-empty argv, client_wrapper.py
calls client/cli/commands/'s run() instead of the chat TUI. This dispatches
straight to shared.executioner.tool_executor.manager (a QemuManager instance)
for commands like list/status/launch/stop/config/resize/clone/snapshot/
delete/network/profiles/templates/system/isos/show-cmd/setup-done/fetch/
bundle — no Ollama, no sanitizer, no preflight, no HTTP call to the
orchestrator for the VM operation itself (this only works when QEMU/the
executor package are locally installed; commands_helpers.py's
_require_manager() errors out cleanly if not). This is the same "thin local
QEMU CLI" mentioned in the GUIDE — it operates on a locally-fetched VM's disk
and config directly, bypassing the AI/gating stack entirely. The one
exception is `launch`, which checks whether the same VM name also exists on
the configured SERVER_URL (via a direct POST /execute call) and asks the user
whether to launch the local or the remote copy if both exist.

────────────────────────────────────────────────────────────────────────────────
3. WORKFLOW: CREATING A VM
────────────────────────────────────────────────────────────────────────────────

User says: "create a Dell G15 laptop VM called gaming-rig with 16GB RAM, stealth"

1. Ollama maps this to:
   tool="create_vm", args={"name":"gaming-rig","memory_mb":16384,
     "manufacturer":"Dell Inc.","product_name":"G15 5520","stealth":True}

2. Sanitizer (local mode, inside orchestrator/pipeline.py): memory_mb cap
   check (16384 ≤ 95% of host RAM → ok), name check (no placeholder), OS type
   defaulted to "linux"

3. Profile auto-detection in tool_executor._execute_create_vm(): "Dell" +
   "G15 5520" matches profile "dell_g15_5520" by product-name token (never by
   manufacturer alone) → apply_profile() adds CPU, display, audio defaults

4. Preflight: product lookup ("Dell G15 5520" → real product via DuckDuckGo, ok)

5. MachineConfig is built with all fields from args + profile defaults

6. stealth=True → hardened=True auto-set, machine_type="q35", gpu forced to
   "none" (disguised as vmware-svga/std VGA by the arg builder) unless the
   caller explicitly asked for a GPU

7. manager.create_vm(cfg):
   - Creates ~/.qemu_vms/gaming-rig/
   - Runs: qemu-img create -f qcow2 disk0.qcow2 40G
   - Copies OVMF_VARS_4M.fd → OVMF_VARS.fd
   - Writes config.json
   - Because stealth=True: calls generate_guest_setup("gaming-rig")
     → writes guest_setup.sh in the VM directory

8. Result: {"success":True, "name":"gaming-rig", "vm_dir":"~/.qemu_vms/gaming-rig"}

9. Display: green "✓ VM 'gaming-rig' created at ~/.qemu_vms/gaming-rig"
   + dim note: "Stealth guest setup script ready — will prompt automatically on first launch"

────────────────────────────────────────────────────────────────────────────────
4. WORKFLOW: LAUNCHING A VM (SINGLE-MACHINE MODE)
────────────────────────────────────────────────────────────────────────────────

User says: "launch gaming-rig"

1. Ollama: tool="launch_vm", args={"name":"gaming-rig"}

2. http_chat.py's inline gates: context assistant ok, no OS-type guard applies,
   preflight (_preflight_check) confirms the VM exists, no safety confirmation
   configured for launch_vm beyond the ordinary y/n (auto-batched if this tool
   type was already confirmed this turn)

3. execute_tool() (orchestrator/executor_client.py) → API_URL=="local" →
   orchestrator/pipeline.py: sanitiser no-ops (name already clean), name
   resolution matches "gaming-rig" exactly, gate_check passes (name present)

4. tool_executor._run() dispatches to manager.launch_vm("gaming-rig")

5. manager.launch_vm() [executor/api/_vm_runtime.py]:
   a. Loads ~/.qemu_vms/gaming-rig/config.json
   b. Checks if already running (tracked process or a matching PID found on
      the host) — returns {"already_running": True} if so
   c. QemuArgBuilder(config).build() → full QEMU command list
   d. subprocess.Popen(["qemu-system-x86_64", ...], start_new_session=True,
      stdout/stderr → launch.log)
   e. 0.3s liveness check — if QEMU already exited, returns the launch.log
      tail as the error instead of a false "success"
   f. Records PID in memory + persisted state
   g. display=="sdl"/"gtk" (local): QEMU already opened its own window — done,
      no VNC step

6. Display in the TUI: chat_client.py's _render_tool_result() for "launch_vm"
   shows "✓ VNC: host:port" only when display=="vnc"; for sdl/gtk it shows
   the plain success message since the window opened directly on the
   executor's own display, not the client's.

────────────────────────────────────────────────────────────────────────────────
5. WORKFLOW: LAUNCHING A VM (SPLIT MODE, VNC)
────────────────────────────────────────────────────────────────────────────────

User is on their laptop; the orchestrator and/or executor run elsewhere.
SERVER_URL=http://localhost:8080 (an SSH tunnel makes the orchestrator appear
local); the orchestrator's own connection_config.json points API_URL at the
executor, e.g. http://executor-host:8001.

1. Same STEPS 1–5 as section 2 (client → /chat → Ollama → gates) — the
   orchestrator always forces launch_vm's display to "vnc" and
   vnc_bind_local=True when the caller isn't itself local (both
   executor_client.execute_tool() and, redundantly, api_server.py's /execute
   handler apply this override; SDL/GTK cannot render over the network)

2. executor_client.py: API_URL != "local" → POSTs
   {"tool_name":"launch_vm","args":{"name":"gaming-rig","display":"vnc",
    "vnc_bind_local":true},"verbose":false}
   to {API_URL}/execute with "Authorization: Bearer {EXECUTOR_TOKEN}"

3. executor/server.py on the executor machine: token check → tool_name in
   _KNOWN_TOOLS → dispatch_tool() → tool_executor._run() → manager.launch_vm()

4. manager.launch_vm() runs on the executor machine:
   a. Builds QEMU args with vnc bound to 127.0.0.1 only (vnc_bind_local)
   b. Starts QEMU, records PID
   c. Connects to the QMP socket (up to 15 × 0.5s retries)
   d. QMP set_password("vnc", "<8-char random>", connected="keep")
   e. QMP expire_password("vnc", now + 4 hours)
   f. Returns {"success":True, "name":"gaming-rig", "display":"vnc",
              "vnc_port":5901, "vnc_password":"aB3xR9qZ"}

5. executor/server.py logs the call locally and returns the result to the
   orchestrator's HTTP POST.

6. Back in orchestrator/executor_client.py's execute_tool(): the result
   passes through unmodified (open_display's host-patching logic doesn't
   apply to launch_vm) and flows up through http_chat.py to api_server.py's
   response body.

7. chat_client.py's _process_response() renders the VNC line and calls
   _try_open_vnc(port) — tries vncviewer/tigervncviewer/xtigervncviewer/
   gvncviewer/vinagre in order, launching whichever is installed, pointed at
   the SERVER_URL's own host (or "localhost" if SERVER_URL's hostname is
   localhost/127.0.0.1 — the assumption being an SSH tunnel is already
   forwarding that port).

8. If no viewer auto-launches: the TUI prints "vncviewer <host>:<port>" for
   the user to run manually, plus the one-time password shown in the tool
   result (chat_turn.py's REPL path renders a dedicated VNC connection panel
   via shared/display.py's render_vnc_connect(); the HTTP/TUI path renders it
   inline via _render_tool_result()).

────────────────────────────────────────────────────────────────────────────────
6. WORKFLOW: REVERTING AN ACTION
────────────────────────────────────────────────────────────────────────────────

After any reversible operation (create_vm, launch_vm, stop_vm, snapshot_create,
etc.), shared/executioner/tool_executor.py's _run() stores the inverse action
in the module-level _last_revert_action:
  {"tool": "delete_vm", "args": {"name":"gaming-rig"}, "description":"undo create_vm 'gaming-rig'"}

User says: "undo that" or "revert"

1. Ollama: tool="revert", args={}

2. tool_executor._run() handles "revert":
   a. Checks _last_revert_action is not empty
   b. Prints "↩ Revert: undo create_vm 'gaming-rig'"
   c. Requires an interactive terminal to confirm (sys.stdin.isatty()) — in
      the HTTP/split path this means revert only works meaningfully when the
      dispatch happens somewhere with a real TTY attached (local-mode
      chat_loop REPL); over plain HTTP without a TTY it reports "Revert
      cancelled: not running interactively."
   d. If confirmed: recursively calls _run() with the stored inverse tool/args
   e. Clears _last_revert_action so "undo" can't be undone again

Key limitation: only one level of undo. If you do A then B, "undo" undoes B.
Any tool call not in _REVERT_AWARE_TOOLS clears the pending revert action.

Irreversible tools (resize_disk, snapshot_restore, snapshot_delete, delete_vm)
explicitly call _clear_revert() so the previous revert action is forgotten —
these cannot safely define an inverse.

────────────────────────────────────────────────────────────────────────────────
7. WORKFLOW: THE SANITIZER IN DETAIL
────────────────────────────────────────────────────────────────────────────────

orchestrator/sanitizer/sanitizer.py's _sanitise_args() is the safety net that
catches bad AI output before it reaches preflight or execution. It never
blocks — it always returns something. It runs ONLY inside
orchestrator/pipeline.py, i.e. only on the local-mode dispatch path (see
section 2A for what stands in for it in split mode).

Input: tool_name="create_vm", args from Ollama (may be malformed)
Output: cleaned args dict

The sanitiser makes these decisions, in order (config values live in
orchestrator/sanitizer/config.json):

  1. Type coercion
     Int fields strip "GB"/"MB"/"gb"/"mb" suffixes before int(); bool fields
     coerce "true"/"yes"/"1"/"on" strings to real booleans.

  2. VM name cleaning
     Non-alphanumeric characters (besides _/-) are replaced with "_"; if the
     result matches PLACEHOLDER_VM_NAMES (windows-vm, linux-vm, my-vm, vm,
     unnamed, virtual-machine, ... and their underscore variants) it's
     cleared to "" (triggers a clarify downstream).

  3. machine_type cleaning
     If machine_type looks like a profile name rather than a real QEMU
     machine type, it's moved into the "profile" field and machine_type is
     dropped.

  4. OS type normalisation
     Using OS_TYPE_ALIASES: "ubuntu"/"mint"/"kali"/"debian" → "linux";
     "win10"/"win11"/"windows11" → "windows"; "osx"/"darwin"/"mac" → "macos".

  5. Enum field validation
     display / gpu / audio / network_mode / disk_format / bios / machine_arch
     / machine_class / os_type are each checked against a valid-value set;
     anything not in the set is replaced with its configured default.

  6. Path fields
     _fix_path() rejects literal placeholder patterns ("<iso>", "/path/to/...",
     etc.), fixes a wrong /home/<user>/ or /Users/<user>/ prefix to the real
     home directory, and converts Windows-style "C:\" paths.

  7. ARM/x86 consistency
     ARM machine types/archs/binary names → kvm=False, hugepages=False.
     ARM CPU model requested on an x86_64 VM → cpu_model="host".

  8. MAC address validation
     Anything not matching a clean 6-octet colon-separated MAC is dropped.

  9. Bridge interface cleaning
     Raw physical NIC names (eth0, wlan0, enp3s0, ...) or known-bad values
     are replaced with the configured default bridge (virbr0).

  10. Resource caps
      memory_mb: clamped to [512, max(host_ram_mb × 0.95, 4096)].
      cpu_cores / cpu_threads: clamped to [1, host logical CPU count].
      disk_size_gb: clamped to [8, 2048].
      Port fields: dropped if outside [1024, 65535].

  11. extra_args / snapshot / network / profile name fields
      extra_args coerced to a list of short strings; snapshot/network/profile
      names stripped to alphanumeric+_/-.

  12. Text fields
      Fields that shouldn't contain path-like content have "/", "\", "`",
      "$(" characters stripped.

  13. Empty optional fields
      Any optional field left None or "" is removed from the dict entirely.

orchestrator/sanitizer/context_gate.py's gate_check() runs immediately after,
also only inside pipeline.py: it checks a per-tool required-field table
(context_gate_config.json) and returns a clarify dict listing every missing
field at once if any required field is still absent (e.g. create_vm requires
name and os_type).

8. WORKFLOW: STEALTH VM SETUP (END-TO-END)
────────────────────────────────────────────────────────────────────────────────

This is the full end-to-end process for creating a VM that passes
hardware fingerprinting checks.

PHASE 1 — CREATE (one command)
  User: "create a Dell Latitude 5530 VM called work-laptop, stealth, hardened,
         8GB RAM, serial number 3K8LP52"

  Result:
  - ~/.qemu_vms/work-laptop/ created
  - config.json has: stealth=True, hardened=True, manufacturer="Dell Inc.",
    product_name="Latitude 5530", serial_number="3K8LP52", machine_class="laptop"
  - guest_setup.sh written to vm dir
  - smbios_chassis.bin written to vm dir (chassis type byte = 9)

PHASE 2 — INSTALL OS
  User: "launch work-laptop with the ISO"
  → QEMU starts with the ISO attached
  → User installs the OS (e.g. Linux Mint) normally inside the VM

PHASE 3 — RUN GUEST SETUP (inside the VM, once)
  User (inside VM):
    curl http://10.0.2.2:8080/guest_setup.sh | bash

  The script:
  1. Writes /etc/modprobe.d/blacklist-qemu.conf
     → blacklists: qemu_fw_cfg, cirrus_qemu
  2. Rebuilds initramfs (detects: update-initramfs / mkinitcpio / dracut)
     → after reboot, these modules will not load
  3. Creates ~/.mozilla/firefox/stealth/user.js
     → WebGL renderer: "Intel(R) Iris(R) Xe Graphics"
     → WebGL vendor: "Intel"
  4. Creates ~/Desktop/stealth-browser.sh
     → launches Firefox with the stealth profile
  5. Writes /usr/local/bin/lspci (wrapper script)
     → calls /usr/local/bin/lspci.real and replaces VMware SVGA output
  6. Writes /usr/local/bin/lsmod (wrapper script)
     → calls /usr/local/bin/lsmod.real and filters qemu* modules

PHASE 4 — REBOOT (inside VM)

PHASE 5 — VERIFY (inside VM)
  lsmod | grep qemu                       → empty
  lspci | grep VGA                        → Intel Corporation ... Iris Xe
  cat /sys/class/dmi/id/chassis_type      → 9
  cat /sys/class/dmi/id/sys_vendor        → Dell Inc.
  inxi -F                                 → Type: Laptop  System: Dell Inc.

PHASE 6 — MARK DONE (on the host)
  gorgon setup-done work-laptop
  (calls manager.mark_stealth_done(), which touches
   ~/.qemu_vms/work-laptop/.stealth_done)
  → HTTP setup-script server will not start on subsequent launches
  → VM is ready for normal use

QEMU LAUNCH FLAGS (what stealth adds to the QEMU command)
  Built by executor/api/qemu_arg_builder.py from the persisted MachineConfig:
    -machine type=q35,acpi_table=<smbios_chassis.bin path>
    -device nec-usb-xhci,id=usb0         (real NEC USB controller)
    -device vmware-svga                   (Linux) or -vga std (Windows)
    -cpu <model>,kvm=off,-vmx,-hypervisor (hardened mode: hide KVM from guest)
    -smbios type=0,vendor=Dell Inc.,version=1.15.0
    -smbios type=1,manufacturer=Dell Inc.,product=Latitude 5530,serial=3K8LP52
    -smbios type=2,product=0H34YX
    -smbios file=<smbios_chassis.bin>     (chassis type = 9)

9. WORKFLOW: SPLIT-MODE SETUP (SEPARATE MACHINES)
────────────────────────────────────────────────────────────────────────────────

Scenario: the orchestrator (AI) and executor (QEMU) run on different
machines, and/or a laptop runs only the client and talks to a remote
orchestrator over SSH tunnel or TLS.

ON THE EXECUTOR MACHINE (owns QEMU/KVM):
  bash files/complementary/install_executor.sh
  Installs: qemu-kvm, qemu-utils, ovmf, socat, libguestfs-tools
  Generates an EXECUTOR_TOKEN, saves it to ~/.gorgon-executor.token (0600)
  Writes a start script and a "gorgon-executor" alias
  Starts: uvicorn executor.server:app --host 0.0.0.0 --port 8001

ON THE ORCHESTRATOR MACHINE (AI + gating):
  bash files/complementary/install_orchestrator.sh
  Installs Ollama + qwen2.5:7b, the orchestrator's Python deps
  Prompts for the executor's URL/token and writes them into
    orchestrator/connection_config.json ("url": "http://executor-host:8001")
  Generates its own client-facing API_TOKEN, saves it to ~/.gorgon.token (0600)
  Writes a "gorgon-serve" alias and (if this machine also acts as the client
    host) a "gorgon" alias pointing client_wrapper.py at itself
  Starts: uvicorn orchestrator.http.api_server:app --host 0.0.0.0 --port 8080

ON THE CLIENT MACHINE (thin remote UI, e.g. a laptop):
  bash files/complementary/setup_client.sh
  Installs only requests + rich (no QEMU, no Ollama)
  Writes to ~/.zshrc:
    export SERVER_URL="http://localhost:8080"   # via SSH tunnel, or the LAN/public IP
    export API_TOKEN="<orchestrator's token>"
    alias gorgon='PYTHONPATH=~/path/to/gorgon/files python3 ~/path/to/gorgon/files/client/client_wrapper.py'

EVERY SESSION (three-machine split):
  1. Open the SSH tunnel(s) if using tunnelled localhost URLs, e.g.:
       ssh -N -L 8080:127.0.0.1:8080 -L 5901:127.0.0.1:5901 \
           you@orchestrator-host
  2. Start chat: gorgon
  3. If a launch returns a VNC panel and no viewer auto-opened:
       vncviewer localhost:5901

10. WORKFLOW: SWITCHING TOPOLOGY
────────────────────────────────────────────────────────────────────────────────

Two independent switches control where things run:

  SERVER_URL (client/config/connection_config.json, or the SERVER_URL env var)
    — where the client's "gorgon" alias sends /chat and /execute. Set it to
    "http://localhost:8080" for a co-located orchestrator (with autostart),
    or to a tunnelled/LAN/public URL for a remote orchestrator.

  API_URL (orchestrator/connection_config.json's "url", or the API_URL env
  var on the orchestrator machine)
    — where the orchestrator's executor_client.py sends tool calls.
    "local" (default) → in-process, no network, executor code must be
      installed on the orchestrator's own machine.
    "http://executor-host:8001" → HTTP to a standalone executor/server.py
      instance on another machine.

These two switches are independent: you can have a remote client talking to
a co-located orchestrator+executor, or a local client talking to a fully
split orchestrator+executor, or any combination. No code change is needed —
executor_client.py's execute_tool() branches on API_URL automatically, and
chat_client.py's _is_localhost()/autostart logic only ever concerns itself
with SERVER_URL.

────────────────────────────────────────────────────────────────────────────────
11. WORKFLOW: TOKEN MANAGEMENT
────────────────────────────────────────────────────────────────────────────────

TWO SEPARATE TOKENS
  API_TOKEN     — the orchestrator's client-facing secret (checked by
                  orchestrator/http/api_server.py's _require_token on every
                  endpoint except /health; localhost callers are always
                  exempt). Precedence: API_TOKEN env var → ~/.gorgon.token file
                  → server refuses to start.
  EXECUTOR_TOKEN — the executor's own secret, used only between the
                  orchestrator and a split-mode executor/server.py.
                  Precedence: EXECUTOR_TOKEN env var → ~/.gorgon-executor.token
                  → executor/config.json's "token" field.
  These are deliberately never the same variable/value — the orchestrator's
  own incoming client-facing token would almost never match a
  separately-generated executor secret, and orchestrator/executor_client.py
  is explicit that it reads EXECUTOR_TOKEN, not API_TOKEN, for this reason.

HOW A TOKEN IS CHECKED (either one)
  Bearer header: "Authorization: Bearer <token>"
  Constant-time compare via secrets.compare_digest() against the loaded
  token; 401 on mismatch or absence (for non-localhost callers).

TOKEN ROTATION (orchestrator's API_TOKEN, while the server is running)
  POST /rotate-token with the CURRENT token in Authorization:
    curl -X POST http://orchestrator-host:8080/rotate-token \
         -H "Authorization: Bearer currenttoken" \
         -d '{"new_token": "newlongsecrettoken"}'
  Server:
  - Validates new_token length ≥ min_token_length (config.json, default 16)
  - Sets the in-memory token and os.environ["API_TOKEN"]
  - Writes the new token to ~/.gorgon.token (created 0600 from the start —
    no world-readable window)
  - Effect is immediate — no restart needed
  Every client using this orchestrator must update its own API_TOKEN to match.
  There is no equivalent /rotate-token endpoint on the executor — rotating
  EXECUTOR_TOKEN means updating both ~/.gorgon-executor.token on the executor
  and orchestrator/connection_config.json's "token" field, then restarting
  the executor server (and the orchestrator if it cached the old value).
