================================================================================
  GORGON  —  CONFIGURATION REFERENCE  (handbook part 4)
================================================================================

Every configuration file, every key, every environment variable, and every
default value. Where env vars override config keys, that is documented too.

Layout: orchestrator/ (AI + gating + client-facing HTTP API, port 8080),
executor/ (owns QEMU/KVM, optional standalone HTTP service on port 8001),
client/ (thin remote UI), shared/ (code used by more than one tier),
admin/ (standalone curses admin dashboard).

────────────────────────────────────────────────────────────────────────────────
1. ENVIRONMENT VARIABLES
────────────────────────────────────────────────────────────────────────────────

All env vars are read at import time (module level). They override the
corresponding config.json values. Precedence is: env var → config.json key
→ hardcoded fallback in the reading module (shown below as "Default").

SERVER_URL
  Default:  "http://localhost:8080"  (from client/config/connection_config.json →
            server_url, and admin/config/connection_config.json → orchestrator_url)
  Values:   "http://host:port" | "https://host:port"
  Effect:   URL of the gorgon orchestrator that the chat client / admin TUI
            connects to.
  Read by:  client/config, admin/config

API_URL
  Default:  "local"  (from orchestrator/connection_config.json → url)
  Values:   "local" | "http://host:8001" | "https://host:8001"
  Effect:   Read by the ORCHESTRATOR, NOT the client. "local" means the
            orchestrator calls the executor in-process (co-located, single
            machine). "http://host:8001" means the executor runs as its own
            HTTP service (executor/server.py) on a separate machine — split
            deployment.
  Read by:  orchestrator/executor_client.py

API_TOKEN
  Default:  "" (from orchestrator/connection_config.json → token)
  Values:   Any string (recommended: 32+ random chars)
  Effect:   The CLIENT-FACING shared secret checked by the orchestrator's
            HTTP API. Must match between the orchestrator and whatever calls
            /chat or /execute — a client, curl, the admin TUI, etc.
            This is a DIFFERENT secret from EXECUTOR_TOKEN below. Conflating
            the two is a real footgun: orchestrator/executor_client.py has
            a comment explicitly warning against reusing API_TOKEN as the
            executor secret, because doing so would make the orchestrator
            send its client-facing token to the executor, which almost never
            matches the executor's independently configured secret.
            The orchestrator refuses to start without a token: it checks
            API_TOKEN first, then falls back to ~/.gorgon.token (file). If
            neither is set, orchestrator/http/api_server.py exits with an
            error at startup.
  Read by:  client/config (sent in Authorization header),
            admin/config (sent in Authorization header),
            orchestrator/http/api_server.py (compared against incoming tokens)
  Also read from: ~/.gorgon.token (file fallback, orchestrator side only)

EXECUTOR_TOKEN
  Default:  "" (from orchestrator/connection_config.json → token, when read
            by the orchestrator; from executor/config.json → token, when
            read by the executor itself)
  Values:   Any string (recommended: 32+ random chars)
  Effect:   A SEPARATE secret from API_TOKEN. Read by the ORCHESTRATOR and
            used only when API_URL points at a remote executor (split mode)
            — it's what the orchestrator presents to the EXECUTOR's own
            HTTP API. Ignored entirely when API_URL is "local" (no HTTP hop
            to authenticate).
            On the executor side, executor/server.py loads its own copy of
            this token with precedence: EXECUTOR_TOKEN env var →
            ~/.gorgon-executor.token (file) → "token" key in
            executor/config.json. If none of the three yield a token, the
            executor prints a startup warning and refuses all remote
            (non-localhost) connections.
  Read by:  orchestrator/executor_client.py (orchestrator side),
            executor/server.py (executor side, independently)

API_TIMEOUT
  Default:  120 (from connection_config.json → timeout, on whichever side
            is reading it)
  Values:   Integer (seconds)
  Effect:   HTTP request timeout for /chat, /execute, and executor calls.
  Read by:  client/config, orchestrator/executor_client.py

API_CA_CERT
  Default:  None
  Values:   Absolute path to a PEM-format CA certificate file
  Effect:   When set, the reader uses this CA to verify the remote TLS
            cert. Used for self-signed certificates. When unset, the system
            CA bundle is used (correct for Let's Encrypt).
  Read by:  client/config, orchestrator/executor_client.py

API_VERIFY_SSL
  Default:  "1" (verify enabled)
  Values:   "1" (verify) | "0" (skip verification)
  Effect:   "0" disables TLS certificate verification entirely.
            NEVER use in production — only for local development.
  Read by:  client/config, orchestrator/executor_client.py

OLLAMA_MODEL
  Default:  "qwen2.5:7b" (from orchestrator/ai/config.json → ollama.model)
  Values:   Any Ollama model name ("llama3.1:8b", "mistral", etc.)
  Effect:   Which model is used for all AI calls.
  Read by:  orchestrator/ai/ollama_client.py

OLLAMA_URL
  Default:  "http://localhost:11434" (from orchestrator/ai/config.json →
            ollama.url)
  Values:   Any URL where Ollama is reachable
  Effect:   Where the Ollama API server is. Always orchestrator-local —
            Ollama is never split out to the executor machine.
  Read by:  orchestrator/ai/ollama_client.py

    # split deployment. Overridden by API_URL env var.

    # Minimum accepted length for API_TOKEN. Shorter tokens are rejected
    # at orchestrator startup.

  "ca_cert": "",
    # Path to custom CA cert (empty = use system bundle).

  "client_allowed_vms": [],
    # VMs visible and accessible to clients. Empty = all VMs allowed.
    # Non-empty = allowlist. Hidden VMs return "not found" to clients (opsec).

  "client_allowed_profiles": [],
    # Hardware profiles visible to clients. Empty = all profiles allowed.

  "allowed_remote_tools": [],
    # Tools the HTTP API accepts from remote clients. Any tool not in this
    # list returns 403 Forbidden before preflight runs. Enforced at
    # executor_client level so it covers both /execute calls and
    # AI-initiated /chat calls. Empty list here = every remote tool call is
    # blocked — populate it before exposing the orchestrator remotely.
    # send_monitor_cmd should stay excluded (raw QEMU monitor access from an
    # unauthenticated remote connection is a security risk).

  "session_ttl_seconds": 3600,
    # How long an idle chat session is kept before eviction.

  "max_sessions": 1000,
    # Maximum concurrent chat sessions held in memory.

  "max_message_length": 32768
    # Maximum accepted length (characters) of a single /chat message body.
}

ACCESS CONTROL ENFORCEMENT:
  Two-layer enforcement — clients cannot bypass either layer:
  1. GET /sync filters the VM/profile inventory before the client sees it.
  2. executor_client.execute_tool() blocks tool calls targeting hidden VMs
     AND tools not in allowed_remote_tools, for both /execute and /chat paths.

4. orchestrator/ai/context_assistant_config.json
   (read by orchestrator/ai/context_assistant.py)
────────────────────────────────────────────────────────────────────────────────

Despite the name, this is NOT a set of message templates + conditions — it
drives the context-assistant's tool-hint detection, slot extraction, and
contradiction/hallucination checks run against each AI tool call before
preflight.

{
  "tool_hints": {
    "create_vm":  ["create", "make", "new vm", "spin up", ...],
    "delete_vm":  ["delete vm", "remove vm", "destroy vm", ...],
    ... one entry per tool name ...
    # Phrases in the user's message that suggest a given tool is the right
    # one. Used to flag "tool_mismatch" when the AI picks a different tool.
  },

  "slot_patterns": {
    "name":        ["\\bcalled ([\\w][\\w\\-_\\.]*)", "\\bnamed (...)", ...],
    "snap_name":    ["\\bsnapshot (...)", "\\bsnap (...)", ...],

    "new_name":     ["\\bto (...)", "\\bas (...)", ...]

  name             str    "vm-default" [cfg]  VM directory name, used everywhere

  hostname         str    "localhost"  [cfg]  guest hostname hint

  uefi             bool   True         [cfg]  enables OVMF firmware path (forced True if bios startswith ovmf and OVMF is available)
  uefi_vars        str|None None       [hc]   path to this VM's persistent OVMF_VARS copy
  cpu_model        str    "host"       [cfg]  -cpu {cpu_model}[,flags]

11. HARDWARE PROFILES (built-in, executor/api/config.json → hardware_profiles)
────────────────────────────────────────────────────────────────────────────────

Built-in profiles are stored inline in executor/api/config.json (loaded by
executor/api/profiles.py as HARDWARE_PROFILES). Custom (user-created)
profiles are saved separately to ~/.qemu_vms/_profiles/*.json and merged in
at runtime by get_all_profiles(); apply_profile() overlays a profile's

  bios_versions      list[str]        realistic BIOS version choices to rotate through

  _notes             str              free-text caveats

13. client/config/CLI_config.json  (+ CLI_config.defaults.json)
────────────────────────────────────────────────────────────────────────────────

Client config lives in the client/config/ package. Every setting + its default
is in CLI_config.defaults.json (the manifest); CLI_config.json holds only the
user's overrides and WINS on merge. The loader (client/config/__init__.py) reads
both and exposes named constants — the chat UI (client/ui/chat_client/) and the
direct CLI (client/cli/) consume those, no longer json.load-ing files directly.
The keys below are the appearance/behaviour ones a user usually overrides; see
CLI_config.defaults.json for the full list (palette colours, glyphs, spinner,
hint line, autostart spawn params, timeouts, paths, VNC port, …).

{
  "text_color": "#7355a3",
    # Hex color for body/dim text in the chat TUI (VM list, system info,
    # hints). Applied via curses.init_color() if the terminal supports
    # color redefinition; falls back to gray (slot 8) otherwise.

  "font_size": 13,
    # Font size hint sent as an xterm escape sequence on TUI startup:
    #   \033]50;xft:Monospace:size=N\007
    # Works in xterm-compatible terminals; silently ignored elsewhere.

  "wrap_width": 120,
    # Column width AI/tool output is wrapped to before rendering.

  "terminal_rows": 44,
  "terminal_cols": 200,
    # Terminal size the TUI resizes itself to on startup (best-effort;
    # depends on terminal emulator support for resize escape sequences).

  "autostart_poll_count": 20,
  "autostart_poll_interval_s": 0.5,
    # When auto-launching a VM at chat startup, how many times (and how
    # far apart) to poll for it to come up before giving up — 20 × 0.5s = 10s max.

  # NOTE: iso_distro_keywords was REMOVED. The client no longer keeps its own
  # ISO→OS keyword table — that was a drifting duplicate of the executor's
  # authoritative data (executor/api/config.json's iso_os_keywords/linux_distros,
  # consumed by infer_os_name). The ISO-vs-declared-OS mismatch check now runs
  # server-side (executor _vm_constants.iso_os_warning), and create_vm returns an
  # `iso_os_warning` field the client just renders. Single source of truth.
}

16. admin/config/admin_config.json  (+ admin_config.defaults.json)
────────────────────────────────────────────────────────────────────────────────

Same defaults+overrides model. admin_config.defaults.json is the full manifest
(~60 keys: palette, glyphs, timeouts, server-spawn params, AND every dashboard

{
