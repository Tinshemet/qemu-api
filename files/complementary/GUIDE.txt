1.1  ONE-COMMAND INSTALL
─────────────────────────
  cd ~/path/to/qemu-api
  bash install.sh

  Handles automatically:
  - apt packages    qemu-kvm, qemu-utils, ovmf, virt-viewer, socat, cpu-checker
  - KVM group membership
  - Python venv     ~/qemu-env  with requests, psutil, rich, fastapi, uvicorn, httpx
  - Ollama install + qwen2.5:7b model pull
  - systemd user service  (Ollama auto-starts on login)
  - Bridge networking config   /etc/qemu/bridge.conf
  - zsh alias       qemu-api
  - Self-test

  To uninstall:  bash install.sh --uninstall

1.2  MANUAL SETUP
──────────────────
  # On the SERVER machine (AI + QEMU):
  sudo apt install qemu-kvm qemu-utils ovmf socat python3-venv
  PY_VER=$(python3 -c 'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")')
  sudo apt install "python${PY_VER}-venv"
  python3 -m venv ~/qemu-env && source ~/qemu-env/bin/activate
  pip install requests psutil rich fastapi uvicorn httpx
  curl -fsSL https://ollama.com/install.sh | sh
  ollama serve & && ollama pull qwen2.5:7b

  # Start the HTTP API (server machine):
  API_TOKEN=mysecret PYTHONPATH=files uvicorn server.http.api_server:app --host 0.0.0.0 --port 8080
  # Or use the alias:  qemu-api-serve

  # On the CLIENT machine (your laptop — thin UI only):
  sudo apt install python3-venv
  PY_VER=$(python3 -c 'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")')
  sudo apt install "python${PY_VER}-venv"
  python3 -m venv ~/qemu-env && source ~/qemu-env/bin/activate
  pip install requests rich

  Add to ~/.zshrc:
    source ~/qemu-env/bin/activate
    export SERVER_URL=http://localhost:8080   # or server LAN IP
    export API_TOKEN=mysecret
    alias qemu-api='PYTHONPATH=~/path/to/qemu-api/files python3 ~/path/to/qemu-api/files/client/client_wrapper.py'

SECTION 2 — ARCHITECTURE OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every request passes through five protection layers before QEMU sees it:

  User Input
      │
      ▼
  qwen2.5:7b (Ollama)           server/ai/ollama_client.py
  Selects tool, builds args
      │
      ▼
  _sanitise_args()              shared/sanitizer/sanitizer.py
  Type coercion, enum normalisation, path fixing,
  placeholder removal, resource caps, arch checks
      │
      ▼
  _preflight_check()            shared/preflight/validator.py
  + _validate_with_internet     Reality check: ok / auto_fix / ask_user / abort
  + _validate_profile_for_host  QEMU binary check, CPU arch check, product lookup,
                                profile vs host capability check
      │
      ▼
  execute_tool()                shared/executioner/tool_executor.py
  Hard guards: arch mismatch, name validation,
  Windows UEFI enforcement, clarify responses
      │
      ▼
  QemuManager                   shared/api/qemu_manager.py
  Actual QEMU/KVM operations

Even if the AI hallucinates bad args, they are caught and either silently
corrected or the user is asked before anything dangerous happens.

The codebase is split into three canonical directories:
  server/   AI + QEMU engine; exposes /chat and /execute over HTTP
  client/   Thin chat UI (your laptop); optional local QEMU CLI for fetched VMs
  shared/   Used by both: QEMU engine, executioner, sanitizer, preflight, display

In local mode both server and client run on the same machine (server_url=localhost).
In remote mode the server runs on a separate machine; the client connects over HTTPS.

3.1  THE SEAM — server/executor_client.py
─────────────────────────────────────────────
  On the server machine the AI and QEMU engine are co-located, so tool
  execution is always a direct in-process call.  executor_client.py is now
  a thin re-export of shared.executioner.execute_tool.

  The server's HTTP /chat endpoint calls process_message() in cli.py, which
  calls execute_tool() from server/executor_client.py, which calls
  shared/executioner/tool_executor.py directly — no HTTP round-trip.

  The remote split is now:
    client/ui/chat_client.py  →  POST /chat  →  server/http/api_server.py
  rather than:
    server/ai/cli.py  →  POST /execute  →  client/server/api_server.py

3.2  LOCAL MODE (single machine)
──────────────────────────────────
  ┌─────────────────────────────────────────────────────────────┐
  │                      One machine                            │
  │                                                             │
  │  client/ui/chat_client.py  (chat_loop)                     │
  │    ↓ user types a prompt  →  POST /chat  (localhost)        │
  │  server/http/api_server.py                                  │
  │    ↓ calls process_message()                                │
  │  server/ai/ollama_client.py                                 │
  │    ↓ Ollama returns a tool call                             │
  │  shared/sanitizer/sanitizer.py  (_sanitise_args)           │
  │    ↓ args cleaned and normalised                            │
  │  shared/preflight/validator.py  (_preflight_check)         │
  │    ↓ full state check (real VM existence, real disk)        │
  │  shared/executioner/tool_executor.py  (execute_tool)       │
  │    ↓ hard guards, then dispatch                             │
  │  shared/api/qemu_manager.py  (QemuManager)                 │
  │    → runs QEMU/KVM directly                                 │
  └─────────────────────────────────────────────────────────────┘

  The interactive loop (server/ai/cli.py → chat_loop) is also available
  and bypasses HTTP entirely — useful for server-side debugging.

3.3  REMOTE MODE (server + thin client)
────────────────────────────────────────
  ┌──────────────────────────┐            ┌────────────────────────────────────────┐
  │   User's laptop          │            │          Server machine                │
  │   (client)               │            │       (AI + QEMU)                      │
  │                          │            │                                        │
  │  client/ui/chat_client   │            │  server/http/api_server.py             │
  │    ↓ user types a prompt │   HTTPS    │  (FastAPI via uvicorn on port 8080)    │
  │    POST /chat ───────────┼───────────►│                                        │
  │    ← {text, tool_results}│            │   ├─ token check (Bearer header)       │
  │       needs_input?       │            │   ├─ process_message()                 │
  │    → user confirms       │            │   │    Ollama + tool loop              │
  │    POST /chat (confirm)──┼───────────►│   │    sanitizer + preflight           │
  │                          │            │   └─ shared/executioner/tool_executor  │
  │  Rich panels rendered    │            │         → runs QEMU/KVM on server      │
  │  from returned results   │            │                                        │
  │                          │◄───────────┤  GET /health  POST /rotate-token       │
  │                          │            │  GET /images/{vm}  (streaming)         │
  └──────────────────────────┘            └────────────────────────────────────────┘

  Session state (conversation history) lives on the server, keyed by
  session_id. The client needs no Ollama installation.

  When an action requires confirmation (e.g. delete_vm), the server returns
  needs_input with the question. The client shows it with Rich and re-sends
  the user's reply with auto_confirm=True.

3.4  SETUP SCRIPTS — WHICH ONE TO RUN
──────────────────────────────────────
  install.sh          Full local setup. Run this if QEMU and the AI chat
                      are both on the same machine. Does everything.

  setup_server.sh     Server-only setup. Run this on the server machine.
                      Installs Ollama + qwen2.5:7b, QEMU/KVM, Python venv,
                      API token, systemd service (qemu-api HTTP API), Ollama
                      systemd service, SSH server, bridge networking.
                      Works on native Linux and WSL2.
                      Pre-set to skip prompts:
                        API_TOKEN=mytoken bash setup_server.sh

  setup_client.sh     Client-only setup. Run this on the user's laptop.
                      Installs Python venv + Rich + requests only (no QEMU).
                      Prompts for SERVER_URL and API_TOKEN.
                      Writes connection_config.json and adds qemu-api alias.
                      Pre-set to skip prompts:
                        SERVER_URL=http://192.168.1.10:8080 API_TOKEN=x bash setup_client.sh

  setup_wsl2.ps1      Windows-only. Run as Admin after setup_server.sh when
                      the server machine is WSL2 on Windows. Handles port
                      forwarding and the reboot auto-refresh scheduled task.

  Typical remote scenario:
    Server machine (Linux/WSL2):  bash setup_server.sh
    Server machine (WSL2+Win):    .\setup_wsl2.ps1    (as Admin)
    Your laptop:                  bash setup_client.sh

3.5  STARTING REMOTE MODE
──────────────────────────
  On the SERVER machine — start the HTTP API:

    source ~/qemu-env/bin/activate
    cd ~/qemu-api

    # HTTP (SSH tunnel recommended for remote access)
    API_TOKEN=mysecrettoken PYTHONPATH=files uvicorn server.http.api_server:app \
        --host 0.0.0.0 --port 8080

    # HTTPS / TLS (direct access without SSH tunnel)
    API_TOKEN=mysecrettoken PYTHONPATH=files uvicorn server.http.api_server:app \
        --host 0.0.0.0 --port 8443 \
        --ssl-certfile /path/to/cert.pem --ssl-keyfile /path/to/key.pem

  Or use the alias created by setup_server.sh:
    qemu-api-serve

  On the CLIENT machine (your laptop):

    # Open SSH tunnel first (if connecting over the internet):
    ssh -N -L 8080:127.0.0.1:8080 -L 5901:127.0.0.1:5901 user@server-ip

    export SERVER_URL=http://localhost:8080   # or server's LAN IP without tunnel
    export API_TOKEN=mysecrettoken
    qemu-api                                  # opens the Rich chat UI

  With TLS on the client machine:

    export SERVER_URL=https://192.168.1.10:8443
    export API_TOKEN=mysecrettoken
    export API_CA_CERT=/path/to/ca.pem       # custom CA cert (optional)
    qemu-api

3.6  CONFIGURATION REFERENCE
──────────────────────────────
  Each side has its own connection_config.json:

  files/server/connection_config.json  (read by server/executor_client.py):
    {
      "url":        "local",     ← always "local" — server runs QEMU co-located
      "token":      "",          ← API token for /execute endpoint
      "timeout":    120,
      "verify_ssl": true
    }

  files/client/connection_config.json  (read by client/ui/chat_client.py):
    {
      "server_url": "http://localhost:8080",  ← qemu-api server address
      "token":      "",                       ← shared secret (prefer API_TOKEN env)
      "ca_cert":    null,                     ← path to CA cert for self-signed HTTPS
      "verify_ssl": true,
      "timeout":    120
    }

  Port forwards in VM config use the format [host, guest, proto]:
    "port_forwards": [[8080, 8080, "tcp"]]

  Environment variables override config.json:

    SERVER_URL      Server URL for the chat client (default: http://localhost:8080)
    API_TOKEN       Shared secret for API authentication
    API_TIMEOUT     HTTP timeout in seconds (default: 120)
    API_CA_CERT     Path to custom CA certificate for TLS verification
    API_VERIFY_SSL  Set to "0" to skip TLS certificate verification (dev only)

  Token persistence on the server:
    The API server reads the token from ~/.qemu-api.token (chmod 0600).
    API_TOKEN env var takes precedence. The token file survives server restarts.

    Rotate the token while the server is running:
      curl -X POST http://server:8080/rotate-token \
           -H "Authorization: Bearer currenttoken" \
           -H "Content-Type: application/json" \
           -d '{"new_token": "newlongsecrettoken"}'
    The new token is written atomically to ~/.qemu-api.token and takes effect
    immediately without a server restart.

3.7  API ENDPOINTS (server/http/api_server.py)
──────────────────────────────────────────────────
  All endpoints except /health require the Authorization: Bearer <token> header.

  GET  /health
    Returns {"status":"ok"} — no auth required.
    Used by the client for connectivity checks.

  POST /chat                                              ← PRIMARY ENDPOINT
    Body: {"message": str, "session_id": str|null,
           "auto_confirm": bool, "verbose": bool}
    1. Verifies Bearer token.
    2. Loads conversation history from in-memory session store (keyed by session_id).
    3. Calls process_message() — runs full agentic AI loop server-side.
    4. Saves updated conversation history back to session store.
    5. Returns {"session_id": str, "text": str, "tool_results": list,
                "needs_input": {type, question, options, field, proposed}|null}
    When needs_input is non-null, the client shows the prompt and re-sends
    the user's reply (with auto_confirm=True for destructive confirmations).

  DELETE /sessions/{session_id}
    Clears the conversation history for the given session.
    Auth required.

  POST /execute                                           ← DIRECT TOOL CALL
    Body: {"tool_name": str, "args": dict, "verbose": bool}
    1. Verifies Bearer token.
    2. Checks tool_name is in allowed_remote_tools.
    3. Overrides display: sdl/gtk → vnc + vnc_bind_local=True (remote mode).
    4. Runs shared/preflight/validator.py (full, stateless_only=False).
    5. If preflight returns auto_fix, applies the correction.
    6. Calls shared/executioner/tool_executor.execute_tool().
    7. Returns {"result": ..., "preflight": bool, ...}

  GET  /images/{vm_name}
    Streams the VM's primary disk image (disk0.qcow2).
    Supports HTTP Range header for resume on interrupted downloads.
    Returns X-SHA256 header with the precomputed checksum.
    Auth required.

  GET  /images/{vm_name}/sha256
    Returns {"sha256": "abcdef..."} for the VM's primary disk.
    Faster than downloading the full image — use to check if local copy
    is already up to date before downloading.
    Auth required.

  POST /rotate-token
    Body: {"new_token": str}   (must be ≥ 16 characters)
    Replaces the in-memory token and persists to ~/.qemu-api.token.
    Auth required.

3.8  TOOL ALLOWLIST
─────────────────────
  The allowed_remote_tools list in config.json controls which tools can be
  called via the HTTP API. If a tool is not in the list, the server returns
  403 Forbidden before running preflight.

  Default allowed tools:
    create_vm, launch_vm, stop_vm, list_vms, vm_status,
    delete_vm, clone_vm, resize_disk, snapshot, restore_snapshot,
    check_system, list_profiles, check_profile_compatibility,
    fingerprint_vm, check_disk, revert,
    generate_guest_setup, mark_stealth_done,
    create_cluster, delete_cluster, list_clusters

  Tools not in the list (e.g. send_monitor_cmd for raw QEMU monitor access)
  are blocked at the API boundary. Adjust the list to fit your threat model.

3.9  VNC IN REMOTE MODE
─────────────────────────
  When a VM is created or launched via the remote API, the server
  automatically overrides SDL/GTK display to VNC with local binding:

    display = vnc
    vnc_bind_local = True    ← binds to 127.0.0.1, not 0.0.0.0

  After QEMU starts, the server:
  1. Retries the QMP socket connection (15 attempts × 0.5s = 7.5s max).
  2. Calls QMP set_password with a random one-time password (8 chars).
  3. Calls QMP expire_password with a 10-minute absolute timestamp.
  4. Returns vnc_port and vnc_password in the result.

  The provider machine displays a connection panel:
    ┌─────────────────────────────────────────────────────┐
    │  VNC — Tunnel first, then connect                   │
    │  ssh -L 5901:localhost:5901 user@192.168.1.10       │
    │  vncviewer localhost:5901                           │
    │  password: aB3xR9qZ   (expires in 10 minutes)      │
    └─────────────────────────────────────────────────────┘

  The VNC port is never exposed directly to the network. An SSH tunnel is
  the required connection method. The one-time password adds a second layer
  in case the tunnel endpoint is shared.

  If expire_password is not supported by the installed QEMU version,
  the password stays valid until the VM is stopped (graceful degradation).

3.10  LIVENESS MONITOR
──────────────────────
  When API_URL != "local", the chat loop starts a background daemon thread
  that pings GET /health every 30 seconds.

  If the client machine stops responding:
    Yellow warning: "Client machine health check failed (500)"
    Red alert:      "Client machine at http://... is not responding.
                     Check that 'qemu-api serve' is still running."

  The warning appears inline in the chat session without interrupting it.
  The monitor thread is a daemon — it exits automatically when the provider
  process exits.

3.11  FETCH — DOWNLOAD VM DISK
────────────────────────────────
  fetch <vm_name> [--out /destination/dir]

  Downloads the primary disk image of a VM from the client machine to the
  provider machine. Requires remote mode (API_URL must be set).

  Process:
  1. Checks GET /images/{vm_name}/sha256 — if the local file already exists
     and the checksum matches, the download is skipped.
  2. If the local file exists but checksum differs, resumes with HTTP Range
     from the byte offset where the local file ends.
  3. Downloads the stream with a progress indicator.
  4. Verifies the final SHA256 after download completes.

  Example:
    qemu-api fetch win-test
    qemu-api fetch win-test --out /external/backups/

  The disk is saved as <vm_name>.qcow2 in the output directory.
  If the VM is running, the image reflects in-flight state — stop the VM
  first for a consistent disk image.

4.1  -cu  CUSTOM MACHINE MODE
───────────────────────────────
  Usage:  qemu-api -cu

  Starts the AI chat with product verification disabled. Allows you to create
  VMs with any manufacturer/product_name combination — including fictional
  hardware that doesn't exist in the real world.

  What it disables:
  - DuckDuckGo product lookup (manufacturer + product_name no longer verified)
  - Memory plausibility check (no warning if RAM exceeds product's real max)

  What still runs:
  - QEMU machine type and CPU model checks (local binary queries)
  - ARM/x86 architecture consistency checks
  - All other pre-flight validators
  - Sanitiser (still corrects bad args)

  Use cases:
  - Creating test environments with fictional hardware names
  - Research VMs that don't need to match real product specs
  - Avoiding DuckDuckGo API calls in air-gapped environments

  Example:
    qemu-api -cu
    You: create a VM called lab-01 with manufacturer "AcmeCorp" and
         product "SuperServer 9000", 16GB RAM

  The AI's system prompt also notes custom mode is active so it won't
  warn you about unverifiable hardware names.

4.2  -tf  FINGERPRINT REPORT
───────────────────────────────
  Usage:  qemu-api -tf <vmname>

  Read-only analysis of a VM's configuration. Simulates exactly what
  inxi -M -N -C -D -A -G would report from inside the guest OS, then
  checks each field against known VM fingerprint signatures.

  What inxi sections it simulates:
    inxi -M  System manufacturer, product name, BIOS vendor, BIOS version,
             serial number, chassis type (all from SMBIOS/DMI tables)
    inxi -N  MAC address OUI prefix, NIC driver type
    inxi -C  Hypervisor flag in /proc/cpuinfo, CPU model string
    inxi -D  Disk interface name (vda/vdb), disk model string
    inxi -A  Audio chip device ID
    inxi -G  GPU driver type

  Each field is rated:
    clean      — looks like real hardware, not a VM indicator
    detectable — some fingerprinting tools may flag this
    VM tell    — immediately identifies this as a virtual machine

  Output includes:
  - Per-section table matching inxi output format
  - Score: percentage of fields that look like real hardware
  - Recommendations panel listing what to fix first

  Example output:
    inxi -M  System manufacturer  Dell Inc.         clean
             System product        XPS 15 9520       clean
             BIOS vendor           (not set)         detectable
             BIOS version          (not set)         detectable
             Serial number         (not set)         detectable
             Chassis type          (not set)         detectable
    inxi -N  MAC OUI prefix        52:54:00          VM tell
             NIC driver            virtio-net-pci    detectable
    inxi -C  Hypervisor flag       KVM (kvm=True)    detectable
             CPU model string      host (real CPU)   clean
    inxi -D  Disk interface        virtio-blk        detectable
             Disk model string     QEMU HARDDISK     VM tell
    inxi -A  Audio chip            Intel HDA (hda)   detectable
    inxi -G  GPU driver            virtio-gpu        VM tell

    Score: 28% look like real hardware (2 clean · 6 detectable · 5 VM tells)

  This tool INFORMS only — it never modifies the VM config.
  Use it to understand what a detection tool would see inside your VM.

  Known VM tells and how to reduce them:
  - MAC OUI 52:54:00   → set mac_address to a non-QEMU OUI
  - QEMU HARDDISK      → add model= and serial= to extra_args
  - virtio-gpu         → use vga or std display instead
  - virtio-net         → use e1000 network model (less detectable)
  - BIOS vendor/ver    → set bios_vendor and bios_version in config
  - KVM flags          → set kvm=False (loses performance)

  If OVMF is not found, SeaBIOS fallback is applied automatically.

7.2  THE SANITISER  (_sanitise_args)
──────────────────────────────────────
  Runs on every tool call. Silently fixes:

  VM names:
    Spaces → underscores
    Placeholder names cleared (all hyphen AND underscore variants):
      windows-vm, linux-vm, ubuntu-vm, my-vm, new-vm, test-vm,
      vm, myvm, testvm, unnamed, virtual-machine, vm-1, vm-2, etc.

  OS type aliases:
    ubuntu/debian/fedora/mint → linux
    win10/win11/win64/windows11 → windows
    osx/darwin/mac → macos

  Enum normalisation (case-insensitive):
    NAT→nat, BRIDGE→bridge, SDL→sdl, VNC→vnc
    Invalid → safe default: x11→sdl, alsa→hda, wifi→nat

  Machine type:
    Profile names stripped and profile field auto-set:
      dell_g15_5520 → profile=dell_g15_5520 (machine_type removed)
      office_laptop → profile=office_laptop (machine_type removed)
      gaming_desktop → profile=gaming_desktop (machine_type removed)

  CPU models:
    ARM CPUs on x86_64 VMs → 'host':
      cortex-a53, cortex-a72, cortex-a15, neoverse-n1 etc.

  Bridge interfaces:
    eth0, ens33, enp3s0, wlan0 → virbr0

  MAC addresses:
    7-octet MACs → removed

  Resource limits:
    memory_mb: min 512MB, max 95% of host RAM
    cpu_cores: max = host logical core count
    disk_size_gb: min 8GB

  ISO paths:
    scan_isos()[0] literal → removed FIRST (before username fix)
    /home/user/ → real home dir
    /path/to/  → removed
    <path>, <iso>, $HOME → removed

  ARM/raspi enforcement:
    machine_type=raspi* or machine_arch=aarch64/arm → kvm=False, hugepages=False

7.3  THE PRE-FLIGHT VALIDATOR  (_preflight_check)
───────────────────────────────────────────────────
  Four possible actions:
    ok        proceed
    auto_fix  silently correct and proceed
    ask_user  show yellow warning panel, wait for user response
    abort     block tool, feed error message back to AI

  In local mode:
    stateless_only=False — validator has access to real VM state, disk files,
    and the QEMU binary. All checks run.

  In remote mode (provider side):
    stateless_only=True — validator skips checks that require real state
    (VM existence, disk size, QEMU binary). Only shape and logic checks run.
    The client machine re-runs the full check before execution.

  create_vm triggers:
    ask_user  — placeholder VM name
    ask_user  — VM already exists
    ask_user  — ARM64 ISO + x86_64 VM
    ask_user  — profile has blocking compatibility issues
    ask_user  — ISO file not found
    auto_fix  — profile name used as machine_type
    auto_fix  — Windows without UEFI
    auto_fix  — Windows disk < 40GB → 64GB
    auto_fix  — internet validator catches CPU/machine type issues

  launch_vm:  abort if VM doesn't exist
  delete_vm:  ask_user always (confirmation)
  resize_disk: abort if VM doesn't exist OR if shrinking
  monitor_cmd: ask_user for dangerous commands
  snapshot restore/delete: ask_user always

7.4  THE INTERNET VALIDATOR  (_validate_with_internet)
────────────────────────────────────────────────────────
  Non-blocking, 4s timeout, graceful offline skip.
  Skipped entirely when -cu flag is active.

  Local QEMU machine type check (instant):
    qemu-system-x86_64 -machine help

  Local QEMU CPU model check (instant):
    qemu-system-x86_64 -cpu help

  CPU arch consistency (local knowledge, instant):
    ARM cpu on x86_64 → error / x86 cpu on aarch64 → error

  Product verification (DuckDuckGo API, ~1-2s):
    Checks manufacturer + product_name is a real product.
    SKIPPED when -cu flag is active.

  ISO architecture (filename check, instant):
    arm64/Arm64/aarch64 in ISO name + x86_64 VM → blocked

8.3  LAYER 11 — REMOTE SPLIT TESTS
─────────────────────────────────────
  19 tests covering the provider/client HTTP boundary. No Ollama needed.
  Uses FastAPI TestClient (in-process HTTP, no real network).

  Test categories:
    Stateless preflight (provider side):
      Valid create_vm → ok
      Missing required arg → abort
      Non-existent VM (stateless_only=True) → ok (state not checked provider-side)
      Non-existent VM (stateless_only=False) → abort (state checked client-side)

    VNC arg binding:
      vnc_bind_local=True  → -vnc 127.0.0.1:1,password=on
      vnc_bind_local=False → -vnc :1

    HTTP authentication:
      Valid token → 200
      Missing token → 401
      Wrong token → 401

    Server-side display override:
      sdl display → overridden to vnc + vnc_bind_local=True
      gtk display → overridden to vnc + vnc_bind_local=True
      vnc display → vnc_bind_local=True injected

    Preflight routing:
      Preflight auto_fix → correction applied before execute_tool call
      Abort result → returned as structured error (HTTP 200, error in body)
      ask_user result → returned as clarify response (HTTP 200)

    Executor client connection strings:
      Remote mode → result includes ssh_tunnel_cmd, vnc_connect_cmd, vnc_note
      Non-VNC remote → no connection string fields

8.4  LAYER 3 — RANDOMISED PROMPTS
────────────────────────────────────
  Every AI test has a prompt_pool — a list of phrasings that say the same
  thing differently. Each run picks one randomly based on the seed.

  Vagueness levels (1-5):
    1  precise:  "create a Linux VM called dev-box with exactly 4096MB RAM"
    2  normal:   "create a Linux VM called dev-box with 4GB RAM"
    3  casual:   "make me a linux box called dev-box, 4 gigs"
    4  vague:    "I need a dev box, linux, not much RAM"
    5  minimal:  "linux vm, dev-box"

  Change seed with -s to get different prompt variations:
    python3 test_api.py -l 3 -s 999

8.5  LAYER 5 — PROPERTY INVARIANTS
─────────────────────────────────────
  Five invariants checked across 50 random inputs each (500 with --fuzz):

    prop_sanitiser_never_crashes
      _sanitise_args() must never raise an exception for any input combination.

    prop_preflight_never_crashes
      _preflight_check() must never raise, always return dict with 'action' key.

    prop_sanitiser_idempotent
      Running sanitiser twice must give same result as running once.

    prop_placeholders_always_cleared
      All known placeholder names must always produce empty name after sanitising.

    prop_profile_always_auto_set
      All known profile names used as machine_type must always set profile field.

13.1  OVERVIEW
───────────────
  Stealth mode makes a VM pass casual and automated hardware fingerprinting.
  Achieves ~7/10 stealth. Enable with:

    "stealth": true,
    "hardened": true

  What the API does automatically at launch:
    Display device   Linux → vmware-svga (loads vmwgfx, no "qemu" in name)
                     Windows → VGA std (no VMware fingerprint)
    CPUID            Hides -hypervisor bit, kvm=off, -vmx (hardened only)
    ACPI OEM ID      Set to manufacturer name instead of BOCHS/BXPC
    USB controller   nec-usb-xhci (real NEC PCI IDs, not Red Hat/QEMU)
    Guest setup      Auto-serves guest_setup.sh / guest_setup.ps1 via HTTP
                     on port 8080 on first launch. Suppressed after
                     .stealth_done sentinel is created.

13.2  SMBIOS / HARDWARE FINGERPRINT FIELDS
───────────────────────────────────────────
  These fields are set in config.json and map to SMBIOS tables visible via
  inxi -F, dmidecode, and Windows System Information:

    "manufacturer":  "Dell Inc."         inxi: System: Dell Inc.
    "product_name":  "Latitude 5530"     inxi: product: Latitude 5530
    "serial_number": "3K8LP52"           inxi: serial: 3K8LP52
    "board_product": "0H34YX"            SMBIOS type 2 (baseboard)
    "bios_vendor":   "Dell Inc."         inxi: BIOS: Dell Inc.
    "bios_version":  "1.15.0"            inxi: v: 1.15.0
    "machine_class": "laptop"            chassis type byte 9 (Laptop)
    "smbios_type":   "Notebook"

  machine_class / smbios_type chassis mapping:
    laptop / notebook → 9   desktop → 3   server → 17   tablet → 30

  The chassis type byte is injected via a raw binary (smbios_chassis.bin)
  because QEMU's CLI cannot set this byte directly. Linux's DMI scanner
  uses last-write-wins, so the appended binary overrides QEMU's default.

  After setting these, inxi -F inside the guest shows:
    System:    Type: Laptop  System: Dell Inc.  product: Latitude 5530
               serial: 3K8LP52
    BIOS:      Dell Inc.  v: 1.15.0

13.3  LINUX GUEST SETUP  (guest_setup.sh)
──────────────────────────────────────────
  Run once inside the VM after OS installation. Supports Ubuntu, Mint,
  Kali, Arch — any distro with update-initramfs, mkinitcpio, or dracut.

  From inside the VM:
    curl http://10.0.2.2:8080/guest_setup.sh | bash

  Step 1 — Blacklist QEMU kernel modules:
    Writes /etc/modprobe.d/blacklist-qemu.conf:
      blacklist qemu_fw_cfg
      blacklist cirrus_qemu
    Rebuilds initramfs (auto-detects tool). Reboot required.

  Step 2 — Firefox stealth profile:
    Creates ~/.mozilla/firefox/stealth/user.js:
      user_pref("webgl.renderer-string.override", "Intel(R) Iris(R) Xe Graphics");
      user_pref("webgl.vendor-string.override",   "Intel");

  Step 3 — Stealth browser launcher:
    ~/Desktop/stealth-browser.sh — launches Firefox with the stealth profile.
    Auto-detects firefox vs firefox-esr (Kali).

  Step 4 — lspci / lsmod wrappers:
    lspci wrapper: replaces VMware SVGA II → Intel Iris Xe in both standard
                   and -mm (machine-readable) output formats.
    lsmod wrapper: filters any module starting with "qemu".
    Originals preserved as lspci.real / lsmod.real. Idempotent.

  After reboot, verify:
    lsmod | grep qemu                    # empty
    lspci | grep VGA                     # Intel Iris Xe
    cat /sys/class/dmi/id/chassis_type   # 9
    inxi -F                              # Type: Laptop  System: Dell Inc.

  Mark done (suppresses HTTP server on next launch):
    touch ~/.qemu_vms/<name>/.stealth_done

13.6  BRIDGE NETWORKING  (wired LAN, one-time host setup)
──────────────────────────────────────────────────────────
  By default VMs use QEMU NAT (10.0.2.x). For a real LAN IP, create a
  bridge on the host once and attach the wired NIC to it.

  Ubuntu / Debian (netplan):
    # /etc/netplan/01-bridge.yaml
    network:
      version: 2
      ethernets:
        enp3s0:              # check your interface: ip link
          dhcp4: false
      bridges:
        br0:
          interfaces: [enp3s0]
          dhcp4: true
          parameters:
            stp: false
            forward-delay: 0

    sudo netplan apply

  Linux Mint / Debian (/etc/network/interfaces):
    auto br0
    iface br0 inet dhcp
        bridge_ports enp3s0
        bridge_stp off
        bridge_fd 0

    sudo systemctl restart networking

  VM config for bridge mode:
    "networks": [{"mode":"bridge","bridge":"br0","model":"e1000e"}]

  Note: hardened:true forces NAT for non-stealth VMs, but stealth VMs
  are allowed to use bridge so they get a real LAN IP instead of 10.0.2.x.

SECTION 14 — REAL-WORLD EXAMPLE: FRIEND'S MACHINE AS THE QEMU SERVER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Scenario:
    - Your friend has a powerful desktop at their house running Linux.
      This will be the QEMU machine — it runs the VMs.
    - You are on your laptop anywhere (home, coffee shop, etc.) with Ollama.
      This is the AI provider — it runs the chat loop and talks to the server.

  Two connection approaches are covered below:
    A. SSH tunnel (recommended — no open ports, no certs needed)
    B. TLS direct (requires port forwarding on friend's router)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  CROSS-PLATFORM SUPPORT
  ──────────────────────────────────────────────────────────────────────────────

  The two sides of the split have different OS requirements:

    Provider side (your laptop — runs Ollama and the chat loop):
      Linux    FULL support
      macOS    FULL support  (Python and Ollama both run natively on macOS)
      Windows  FULL support  (Python and Ollama both run on Windows;
                              run from Command Prompt or PowerShell)

    Client side (friend's machine — runs QEMU and the VMs):
      Linux    FULL support  (KVM acceleration, all networking features)
      macOS    PARTIAL       (QEMU runs via Homebrew with HVF acceleration
                              instead of KVM; networking and some SMBIOS
                              features may differ; not tested)
      Windows  NOT SUPPORTED (KVM is a Linux kernel module — it does not
                              exist on Windows; QEMU without KVM runs in pure
                              software emulation which is 10-50x slower and
                              impractical for OS-level VMs)

  The most common cross-platform scenario is:
    You on Linux/macOS/Windows (provider)  →  friend on Linux (client)
  This works in all three provider-OS combinations.

  If your friend's machine is Windows:
    Option 1 — WSL2 (easiest to try):
      WSL2 includes a real Linux kernel. On some hardware and Windows
      builds, KVM passthrough is available inside WSL2.
      Inside WSL2 on the friend's machine:
        check:  ls /dev/kvm    <- exists = KVM available in WSL2
        install and run the server exactly as in the Linux instructions
      Caveat: WSL2 networking is NAT'd by Windows. SSH into WSL2 requires
      extra port forwarding from Windows to the WSL2 IP. Use:
        netsh interface portproxy add v4tov4 \
            listenport=22 listenaddress=0.0.0.0 \
            connectport=22 connectaddress=<WSL2 IP>
      Get the WSL2 IP inside WSL2: hostname -I

    Option 2 — Dual boot (most reliable):
      Install Ubuntu or Linux Mint alongside Windows on the friend's machine.
      Boot into Linux when they want to run the QEMU server.
      All features work fully.

    Option 3 — Linux VM under Hyper-V (nested virtualisation):
      Enable Hyper-V on Windows, create a Linux VM, enable nested
      virtualisation for that VM:
        Set-VMProcessor -VMName "LinuxServer" -ExposeVirtualizationExtensions $true
      Run the QEMU API server inside the Linux VM.
      Nested virtualisation adds a performance penalty but VMs will still
      run significantly faster than pure QEMU software emulation.

  Bottom line:
    If you are on Linux Mint and your friend's machine is Windows,
    the cleanest path is WSL2 (if their hardware supports KVM passthrough)
    or dual-booting Linux on their machine. The server code itself is
    Linux-only — it is not a software limitation that can be patched around,
    it is a hardware virtualisation layer that only exists in the Linux kernel.

  ── WSL2 SETUP (friend on Windows, you on Linux/Mac/Windows) ────────────────

  Both scripts handle this end-to-end. The manual steps are here for
  reference if something needs to be fixed or customised.

  Step 1 — Inside WSL2 (friend runs this):
    Check KVM is available:
      ls /dev/kvm           <- must exist
    If missing: wsl --update in PowerShell as Admin, then restart WSL2.

    Clone/copy the project, then:
      bash complementary/setup_client.sh
    The script installs everything, sets up SSH, saves the token, and
    starts the server. It prints the exact commands to paste on your laptop.

  Step 2 — In PowerShell as Admin on Windows (friend runs this):
    .\complementary\setup_wsl2.ps1
    This configures the port forward (Windows 2222 -> WSL2 22), firewall
    rule, and a scheduled task to refresh it on every reboot automatically.
    At the end it prints the ready-to-paste ssh command for your laptop.

  Step 3 — On your laptop:
    Copy the ssh and export commands printed by setup_wsl2.ps1 and run them.
    Example (your friend's output will have their real IP and token):
      ssh -N \
          -L 8080:127.0.0.1:8080 \
          -L 5901:127.0.0.1:5901 \
          -L 5902:127.0.0.1:5902 \
          -p 2222 \
          friendwsluser@203.0.113.42
      export API_URL=http://localhost:8080
      export API_TOKEN=<token from setup_client.sh output>
      qemu-api

    friendwsluser is the Linux username inside WSL2 — may differ from
    the Windows username. setup_wsl2.ps1 prints it automatically.

  WSL2 caveats:
    - WSL2 closes if no processes are running inside it. The API server
      keeps it alive. setup_client.sh auto-starts it from ~/.bashrc.
    - The QEMU API server binds to 127.0.0.1:8080 inside WSL2.
      Windows cannot reach it directly — only via the SSH tunnel.
    - WSL2 VMs use NAT networking (10.0.2.x inside guests).
      Bridge networking requires additional WSL2 network config.
    - The WSL2 IP changes every Windows reboot. setup_wsl2.ps1 creates
      a SYSTEM-level scheduled task that fixes the port proxy automatically.

14.1  APPROACH A — SSH TUNNEL  (recommended)
──────────────────────────────────────────────

  How it works:
    The API server listens on localhost only (no open internet port).
    You create an SSH tunnel from your laptop to your friend's machine.
    The tunnel makes the remote port appear as a local port on your laptop.
    All traffic — API calls and VNC — travels inside the encrypted SSH connection.

  Network diagram:

    Your laptop                       Friend's machine
    ───────────                       ────────────────
    qemu-api chat     ─ SSH tunnel ─► API server :8080 (127.0.0.1 only)
    localhost:8080                    tool_executor → QEMU
    localhost:5901    ─ SSH tunnel ─► VNC :5901 (127.0.0.1 only)
    vncviewer                         (bound by vnc_bind_local=True)

  ── Step 1: Friend's machine setup ──────────────────────────────────────────

  Clone or copy the project files to the friend's machine, then run:
    bash complementary/setup_client.sh

  This single script handles everything automatically:
    - Detects native Linux vs WSL2
    - Checks /dev/kvm (exits early if KVM unavailable in WSL2)
    - Installs all apt packages (qemu-kvm, ovmf, python3-venv, openssh-server, etc.)
    - Adds user to kvm group
    - Creates the Python venv and installs pip packages
    - Prompts for an API token (or generates one securely)
    - Saves the token to ~/.qemu-api.token (chmod 600)
    - Creates ~/start-qemu-api.sh
    - On native Linux: creates and enables a systemd user service with
      loginctl linger (server auto-starts even without login)
    - On WSL2: adds server auto-start to ~/.bashrc
    - Starts the server and health-checks it
    - Prints the exact ssh and export commands to paste on your laptop

  If the friend's machine is Windows / WSL2, they also run (in PowerShell as Admin):
    .\complementary\setup_wsl2.ps1

  This handles the Windows side:
    - Gets the current WSL2 IP automatically
    - Adds port proxy: Windows 2222 -> WSL2 22
    - Adds Windows Firewall rule for port 2222
    - Creates a scheduled task (runs as SYSTEM at startup) that refreshes
      the port proxy after every reboot when the WSL2 IP changes
    - Prints the ready-to-paste ssh + export commands for your laptop

  To uninstall the Windows-side setup:
    .\complementary\setup_wsl2.ps1 -Uninstall

  ── Step 2: Your laptop setup ───────────────────────────────────────────────

  Clone or copy the project files to your laptop, then run:
    bash complementary/setup_client.sh

  The script asks two questions:
    1. SERVER_URL?    → http://localhost:8080
                         (the SSH tunnel makes the remote server appear as
                          localhost on your laptop)
    2. API_TOKEN?     → paste the token printed by setup_server.sh
                         (it is also saved at ~/.qemu-api.token on the server)

  To skip the prompts entirely, pre-set the values as env vars:
    SERVER_URL=http://localhost:8080 API_TOKEN=mytoken bash setup_client.sh

  It then:
    - Installs only Python venv + requests + rich (no QEMU, no Ollama)
    - Writes client/connection_config.json with server_url and token
    - Writes a block to your shell rc with the alias and env vars:
        export SERVER_URL="http://localhost:8080"
        export API_TOKEN="..."
        alias qemu-api='python3 .../client/client_wrapper.py'
    - Pings the server's /health endpoint to verify connectivity
      (warns if the SSH tunnel is not open yet — that is fine at this step)

  After it finishes:
    source ~/.zshrc   (or ~/.bashrc)

  To point at a different server without re-running the script:
    Edit the SERVER_URL line in ~/.zshrc (or ~/.bashrc) between the
    # qemu-api client start / end comments, then: source ~/.zshrc

  ── Step 3: Open the SSH tunnel ─────────────────────────────────────────────

  Run this on your laptop (replace values with real ones):
    ssh -N \
        -L 8080:127.0.0.1:8080 \
        -L 5901:127.0.0.1:5901 \
        -L 5902:127.0.0.1:5902 \
        friendusername@203.0.113.42

    Flags explained:
      -N                          don't open a shell, just forward ports
      -L 8080:127.0.0.1:8080     your laptop:8080 -> friend's machine:8080
      -L 5901:127.0.0.1:5901     your laptop:5901 -> friend's VNC port 5901
      -L 5902:127.0.0.1:5902     extra VNC port if running a second VM
      friendusername              their Linux username on that machine
      203.0.113.42                their public IP (from Step 1)

  This command blocks while the tunnel is open. Leave it running in a
  separate terminal tab.

  To run the tunnel in the background instead:
    ssh -fN \
        -L 8080:127.0.0.1:8080 \
        -L 5901:127.0.0.1:5901 \
        -L 5902:127.0.0.1:5902 \
        friendusername@203.0.113.42

    Kill it later with:  pkill -f "ssh -fN.*203.0.113.42"

  ── Step 4: Your laptop — start the chat ────────────────────────────────────

  In a new terminal tab:
    source ~/qemu-env/bin/activate
    export API_URL=http://localhost:8080    <- tunnel makes remote look local
    export API_TOKEN=choose-a-long-secret-here
    cd ~/qemu-api/files
    qemu-api

  You are now in the AI chat loop. Everything you type is processed locally
  by Ollama, and tool calls are forwarded through the tunnel to the friend's
  QEMU machine.

  ── Step 5: Create and connect to a VM ──────────────────────────────────────

  In the chat:
    You: create a Linux VM called myvm with 4GB RAM and launch it

  The server creates and starts the VM, then returns:

    +---------------------------------------------------------+
    |  VNC — Tunnel first, then connect                       |
    |  ssh -L 5901:localhost:5901 friendusername@203...       |
    |  vncviewer localhost:5901                               |
    |  password: aB3xR9qZ   (expires in 10 minutes)          |
    +---------------------------------------------------------+

  Because you already added -L 5901:127.0.0.1:5901 to your SSH tunnel in
  Step 3, the VNC tunnel is already open. Just run:
    vncviewer localhost:5901

  Enter the password shown in the panel. You are now viewing the VM's
  display running on your friend's machine from your laptop.

  The VNC password expires in 10 minutes. If you need a new one, stop and
  relaunch the VM — a fresh password is generated each time.

  ── Keeping the server running across reboots ────────────────────────────────

  setup_client.sh handles this automatically (systemd service + linger on
  native Linux, ~/.bashrc auto-start on WSL2). If you need to manage it
  manually after the fact:

  Native Linux:
    systemctl --user status qemu-api         <- check status
    systemctl --user restart qemu-api        <- restart
    journalctl --user -u qemu-api -f         <- live logs

  WSL2:
    cat /tmp/qemu-api.log                    <- server logs
    pkill -f "api_server"                    <- stop
    nohup ~/start-qemu-api.sh > /tmp/qemu-api.log 2>&1 &   <- start

14.2  APPROACH B — TLS DIRECT  (no SSH required, needs port forwarding)
─────────────────────────────────────────────────────────────────────────

  How it works:
    The API server listens on all interfaces with TLS.
    The friend's router forwards a port to their machine.
    Your laptop connects directly — no SSH involved.

  Network diagram:

    Your laptop                  Friend's router           Friend's machine
    ───────────                  ───────────────           ────────────────
    qemu-api chat  -- HTTPS --> port 8443 forward ------> API server :8443
    (API_URL=https://...)        (one-time config)         (0.0.0.0:8443, TLS)

  Note on VNC: VNC is still bound to 127.0.0.1 on the friend's machine.
  In this approach you CANNOT connect to VNC directly — you must open a
  separate SSH tunnel just for the VNC port when you need display access.
  This is why Approach A is recommended: the SSH tunnel handles everything.

  ── Step 1: Generate a TLS certificate on the friend's machine ───────────────

  Option 1 — self-signed (fast, requires copying the cert to your laptop):
    openssl req -x509 -newkey rsa:4096 -nodes \
        -keyout server.key -out server.crt -days 365 \
        -subj "/CN=qemu-api"

  Option 2 — Let's Encrypt (requires a domain name pointing at the public IP):
    sudo apt install certbot
    sudo certbot certonly --standalone -d yourdomain.example.com
    # certs land at /etc/letsencrypt/live/yourdomain.example.com/

  ── Step 2: Port forwarding on the friend's router ───────────────────────────

  Log into the router admin panel (usually 192.168.1.1 or 192.168.0.1):
    Protocol:       TCP
    External port:  8443
    Internal IP:    192.168.1.100   <- friend's machine local IP
    Internal port:  8443

  Give the friend's machine a static local IP so the port forward doesn't
  break after a DHCP lease renewal:
    In router: find the machine's MAC address and assign a static DHCP lease.
    Or on the machine: configure a static IP in network settings.

  ── Step 3: Start the server with TLS ───────────────────────────────────────

  On the friend's machine:
    cd ~/qemu-api
    source ~/qemu-env/bin/activate
    API_TOKEN=choose-a-long-secret-here PYTHONPATH=files uvicorn \
        client.server.api_server:app \
        --host 0.0.0.0 --port 8443 \
        --ssl-certfile server.crt \
        --ssl-keyfile  server.key

  ── Step 4: Your laptop — copy the cert (self-signed only) ──────────────────

  If you used a self-signed cert, copy server.crt to your laptop:
    scp friendusername@203.0.113.42:~/qemu-api/files/server.crt ~/qemu-api-ca.crt

  If you used Let's Encrypt, no cert copy is needed — it is publicly trusted.

  ── Step 5: Your laptop — start the chat ────────────────────────────────────

  Self-signed cert:
    export API_URL=https://203.0.113.42:8443
    export API_TOKEN=choose-a-long-secret-here
    export API_CA_CERT=~/qemu-api-ca.crt
    qemu-api

  Let's Encrypt cert:
    export API_URL=https://yourdomain.example.com:8443
    export API_TOKEN=choose-a-long-secret-here
    qemu-api

  ── Step 6: VNC in Approach B ───────────────────────────────────────────────

  VNC is bound to 127.0.0.1 on the friend's machine regardless of approach.
  When a VM is launched the chat shows the connection panel — but unlike
  Approach A you don't already have a VNC tunnel open.

  Open one when you need it:
    ssh -N -L 5901:127.0.0.1:5901 friendusername@203.0.113.42

  Then connect:
    vncviewer localhost:5901

  This is the extra manual step that makes Approach A simpler for regular use.

14.3  CHOOSING BETWEEN A AND B
────────────────────────────────

  Prefer Approach A (SSH tunnel) when:
    - You don't want to configure port forwarding on the router
    - The friend's public IP changes (dynamic DNS not set up)
    - You want the simplest setup with no certificates to manage
    - This is occasional or ad-hoc use
    - Maximum security: zero attack surface, no port open on the internet

  Prefer Approach B (TLS direct) when:
    - The friend's machine runs the server full-time as a permanent setup
    - Multiple people need access (different clients, rotate-token per session)
    - You have a domain name and Let's Encrypt cert
    - You want the API connection to survive even if SSH drops

14.4  TROUBLESHOOTING
──────────────────────

  "Connection refused" on API_URL:
    SSH tunnel approach:
      - Check the tunnel terminal is still open and has not timed out
      - Re-run: ssh -N -L 8080:127.0.0.1:8080 friendusername@203.0.113.42
      - Verify the server is running on the friend's machine:
          systemctl --user status qemu-api
    TLS approach:
      - Check the server is running on the friend's machine
      - Check port 8443 is forwarded on the router
      - Check friend's firewall: sudo ufw status  (allow 8443 if blocked)

  "Invalid API token":
    - API_TOKEN on your laptop must match exactly what the server started with
    - Check for extra spaces or quotes in the export command
    - If using token file: cat ~/.qemu-api.token on friend's machine

  "Client machine is not responding" (liveness alert in chat):
    - The server process crashed or the SSH tunnel dropped
    - SSH tunnel: re-run the ssh -N command in its terminal tab
    - TLS: check systemctl --user status qemu-api on the friend's machine
    - The chat loop stays alive — once the tunnel or server is back,
      the next command will work normally

  VNC password expired:
    - The QMP-injected password is valid for 10 minutes from VM start
    - Stop and relaunch the VM in the chat: "stop myvm" then "launch myvm"
    - A fresh one-time password is generated each launch
    - The SSH tunnel itself stays open — only the VNC password expires

  Can't reach the friend's machine by public IP from inside their network:
    - Most home routers don't support NAT loopback (hairpin NAT)
    - Test port forwarding only from outside their network (use mobile data)
    - Inside their network, use the local IP instead:
        ssh -N -L 8080:127.0.0.1:8080 friendusername@192.168.1.100

  SSH tunnel times out when idle:
    Add keep-alive settings to ~/.ssh/config on your laptop:
      Host 203.0.113.42
          ServerAliveInterval 60
          ServerAliveCountMax 5

  SSH key authentication (avoid typing password every session):
    On your laptop:
      ssh-keygen -t ed25519
      ssh-copy-id friendusername@203.0.113.42
    After this, the tunnel opens without a password prompt.

  Friend's public IP keeps changing (dynamic IP):
    Ask them to install a free dynamic DNS client:
      sudo apt install ddclient
    Or sign up for a free hostname at no-ip.com or duckdns.org.
    Then use the hostname instead of the raw IP:
      ssh -N -L 8080:127.0.0.1:8080 friendusername@friend.duckdns.org
